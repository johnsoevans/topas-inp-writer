# TOPAS INP Writer

TOPAS-Academic is Bruker's diffraction-analysis program. Its input language (the "INP" format) is a macro-driven scripting language for describing samples, refinement objects, and equations, built on nested `{ }` blocks of keyword/value pairs. This skill packages the TOPAS Technical Reference Manual as a set of reference files so you can write correct, idiomatic INP syntax and debug existing scripts without guessing at keyword names or semantics.

This skill was distilled from a full copy of the Technical Reference; it captures the syntax rules and worked examples faithfully, but the reference files are prose-extracted from the manual, so equations that were originally typeset (e.g. as embedded math objects) may render as plain text approximations. When precision matters (e.g. an exact functional form), say so and flag the uncertainty rather than presenting it as verbatim manual text.

## How to use this skill

1. **Identify what the user is actually trying to do** before writing anything — the same keyword can behave differently depending on which data structure (`str`, `hkl_Is`, `xdd_Is`, Pawley, indexing, PDF) it's used in. When starting a brand-new `.inp` from scratch (no existing file to extend) and the run type isn't already obvious from context, ask — see "Starting a new INP file from scratch" below for the actual question set to use.
2. **Check `example_inp_files/README.md` first** for a hand-picked, carefully annotated, real working `.inp` file close to the task — these are curated for clean layout and heavy inline commenting (parametric/sequential multi-pattern Rietveld, variable-temperature phase transitions, fundamental-parameters fits with preferred orientation/anisotropic broadening) and are a better style/structure model than a bare syntax demo. The README describes what each file covers and calls out specific patterns worth reading; open the actual `.inp` file (in this same folder) once the README points you at the right one. If nothing there is close enough, fall back to `references/examples-index.md` for a real, complete INP script close to the task, then resolve and read the actual file via `TOPAS_DIR` (see "Locating your TOPAS installation" below) — this skill does not bundle those example files themselves, only this curated index of what each one demonstrates (organized by folder, tagged with detected topics: Pawley, PDF, stacking faults, rigid bodies, quant, etc.). A real worked example is almost always a better starting point than writing from scratch — copy its structure and adapt names/values/macros to the user's case rather than reinventing the pattern. To actually read one: `python3 scripts/topas_install.py --example <relative-path-from-the-index>` (e.g. `cf/alvo4a.inp`) prints its resolved location on disk, which you then open with the `Read`/`Grep` tools. Most examples (198 of 280) have a matching `.out` file sitting right alongside the `.inp` (resolve that the same way, with a `.out` extension) — read both together to see exactly what refining changes: placeholder values get replaced with refined ones, each carrying an uncertainty via a trailing backtick (e.g. `beq @ 0.19987`_0.00463`). **If `TOPAS_DIR` isn't set or the file doesn't resolve, say so plainly rather than inventing file content** — you can still reason from `examples-index.md`'s topic description and the manual chapters, just without the literal source to copy from; ask the person to set `TOPAS_DIR` if seeing the real file would help.
3. **Read the relevant manual reference file(s) below for the syntax rules themselves** — don't rely on general crystallography knowledge for TOPAS-specific keyword names, attribute lists, or macro behavior. The manual is large; only open the file(s) that match the task instead of reading everything.
4. **Combine both sources**: the manual chapters explain *why* and the exact rules; the worked examples show *how it looks in a real, working file*. Cross-check a worked example against the manual when something in it looks unfamiliar rather than assuming the example is exhaustive or bug-free.
5. **When debugging**, check the common-errors checklist near the end of this file first, then consult `references/01-syntax-and-parameters.md` and `references/02-equation-operators-and-functions.md` for the exact rules being violated, and look for a similar worked example to compare against line-by-line.
6. **State uncertainty plainly.** If a construct isn't covered in the references or examples and you're relying on general inference, say so — don't present a guess as documented behavior.
7. **Before handing over a finished or edited `.inp` file, run it through `scripts/check_inp_syntax.py`** (see the Scripts section below). It catches unbalanced braces/parens and missing-semicolon equations mechanically, and flags identifiers that look like a typo of a real keyword — cheaper and more reliable than eyeballing a long file for these particular mistakes.
8. **When summarizing or reviewing an `.inp`/`.out` file, actively flag `CS_L`/`CS_G` (`csl`/`csg`) values larger than roughly 500** — confirmed directly by the user (TOPAS-Academic's author): `FWHM ∝ 1/value` for both macros, so sensitivity to the true crystallite size fades rapidly past that point, often showing up as a large refined error and a strong `csl`↔`csg` anti-correlation in `C_matrix_normalized` if present. See `references/04-peak-generation-and-peak-type.md` § "CS_L / CS_G: FWHM is proportional to 1/value" for the full reasoning and a real worked example (`test_examples/simple.inp`).

## Starting a new INP file from scratch — clarifying questions

Use this question set whenever a person asks for a brand-new `.inp` file (no existing file to extend/fix) and the run type isn't already clear from what they've said. Lead with the top-level branch (use `AskUserQuestion` if available); only ask the follow-ups for the branch they pick — don't front-load every question at once. Skip anything already answered by their original request.

**1. What kind of run is this?**

| Answer | Covers | Primary reference(s) |
|---|---|---|
| Rietveld refinement (known structure) | Refine atomic coordinates/occupancies/ADPs against a known model | `01`, `03`, `04`, `05` |
| Pawley / Le Bail (cell known, structure unknown or intensity-only) | Extract intensities with only cell+symmetry, no atom positions | `01`, `04` (`hkl_Is`/`xdd_Is` vs `str`) |
| Indexing (cell unknown) | Determine the unit cell from peak positions | `14` |
| PDF — generation | Convert raw reciprocal-space data into G(r) | `08` |
| PDF — refinement | Refine a structure against G(r) you already have | `09` |
| Charge flipping / ab initio structure solution | Solve an unknown structure from single-crystal or powder data, no starting model | `22` |
| Quantitative phase analysis | Weight fractions across multiple known phases | `11` (usually layered onto a Rietveld fit, not standalone) |
| Stacking faults | Layer/fault-sequence modeling | `10` |
| Rigid bodies | Molecular/ionic fragments moved as a unit | `13` (layered onto Rietveld or PDF, not standalone) |
| Deconvolution | Separate instrumental broadening from sample broadening | `07` |
| Energy minimization / molecular dynamics | Structure optimization by potential energy, not diffraction data | `15`, `16` |
| Protein refinement | Charge-flipping or Rietveld at protein scale | `18`, `19` |
| Magnetic structure refinement | Magnetic scattering/ordering | `12` |

If the answer spans two of these (e.g. "index this pattern, then Rietveld-refine the result" or "generate a PDF then refine a structure against it"), say so explicitly and plan for a multi-stage pipeline (see the `pbso4.raw` peak-search → peak-fit → indexing example in this session's history, or the combined `Include_PDF_Generate` template in `08`/`09`) rather than picking one branch and dropping the rest.

**2. Follow-ups by branch:**

- **Rietveld:** Do you have the structure already (CIF, or space group + atom list), or does it need to come from elsewhere first? Single phase or multi-phase? Data source — lab X-ray (which tube: Cu, Co, Mo, Cr...), synchrotron, or neutron (CW or TOF — changes the peak-shape macro family entirely, see `04`)? Full fundamental-parameters instrument description (`axial_conv`, `Radius`, `Slit_Width`, `LP_Factor`) or a simpler empirical peak shape (`TCHZ_Peak_Type`, plain PV)? Anything beyond a plain fit needed — preferred orientation, anisotropic (`hkl`-dependent) broadening, rigid bodies, an amorphous/broad background component? `do_errors` on?
- **Pawley/Le Bail:** Starting lattice parameters and space group — from indexing, from a literature cell, or a rough estimate to refine?
- **Indexing:** Do you already have a peak list (position ± intensity), or does a peak search need to happen first (see the `pbso4.raw` peak-search → peak-fit-refine → index pipeline for the pattern to follow if TOPAS's own Peak-Search hasn't already produced one)? Any prior knowledge to seed the search — approximate max d-spacing, minimum lattice parameter, or a crystal system to prioritize?
- **PDF (either branch):** What real-space r-range and Q_max? Lab or synchrotron/neutron source (affects `dQ_damping`/`convolute_Qmax_Sinc` need — see `08`'s "not in the manual" section)? For refinement specifically: `beq` or `pdf_for_pairs`-based broadening, or ADPs (`ADP_5`/`ADP_7`/`Uij`)?
- **Charge flipping:** Single-crystal or powder data? Is the space group known, or is that being determined too? Protein-scale (see the `cf-protein/` example family)?
- **Quant:** Which phases, and is this layered onto a Rietveld fit that already exists or being built fresh alongside it?
- **Stacking faults:** What layer types and how many; is a specific fault probability/sequence being tested, or searched for?

**3. Common to nearly every branch, ask once the type is settled (not before):** data file path and format (`.raw`, `.xy`, `.xye`, GSAS, ...), radiation/wavelength if not already implied by the data source answer above, and any output needs (`do_errors`, CIF/pdCIF output, plots).

## Locating your TOPAS installation

This skill does not bundle copies of the `.inc`/`.txt` system files (`topas.inc`, `pdf.inc`, the space-group/scattering-factor tables, ...) or the worked example `.inp`/`.out` files — those ship with every TOPAS release already, and keeping a second copy inside the skill meant maintaining two copies of the same content, with the bundled one inevitably going stale relative to the real install (a newer release, a locally-edited `topas.inc`, extra examples added over time).

**Set the `TOPAS_DIR` environment variable to the root of your TOPAS installation** (wherever `tc.exe`/`TA.EXE` live, or any ancestor directory) before using this skill. `scripts/topas_install.py` searches under it (once per session, cached) to resolve:

```
python3 scripts/topas_install.py --inc-dir                  # the real .inc macro library directory
python3 scripts/topas_install.py --example cf/alvo4a.inp    # a specific example file's real path
python3 scripts/topas_install.py --kernel-schema-html        # the pre-rendered "Show Schema" page
python3 scripts/topas_install.py --macro-browser-html        # a generated macro-browser page, if one exists yet
python3 scripts/topas_install.py --technical-reference-pdf   # the manual PDF (used for §21.2 macro descriptions)
python3 scripts/topas_install.py --keyword-tree-html         # a generated keyword-hierarchy page, if one exists yet
```

`scripts/check_inp_syntax.py` and `scripts/expand_inp_macros.py` both call this automatically (no flag needed) and print a one-line note to stderr confirming they found your install. Reading an example file yourself (see step 2 above) means resolving its path through `topas_install.py --example` first, then using `Read`/`Grep` on the printed path.

**Without `TOPAS_DIR` set, this skill has no fallback copy**, which degrades six specific things, cleanly and with a clear message rather than a silent gap:
- `check_inp_syntax.py`'s macro-arity check (needs real macro definitions to check argument counts against) — its other checks (brace/paren balance, missing semicolons, keyword-typo detection, which is sourced from the manual chapters below, not the `.inc` library) are unaffected and keep working normally.
- `expand_inp_macros.py`'s expansion of library macros (`LP_Factor`, `CS_L`, everything in `topas.inc`/`pdf.inc`) — macros defined in the file being expanded itself still expand fine.
- Reading the literal content of a worked example referenced in `references/examples-index.md` — the index's own topic descriptions remain available either way, just not the source file to copy from.
- "Show Schema" (see the `kernel_structure_tree.html` entry below) — there is no bundled fallback copy at all; only a real TOPAS install's own copy can satisfy this.
- `scripts/generate_macro_browser.py` — can't find the real `.inc` library to harvest macros from and exits immediately (`-o` does NOT work around this — the `.inc`-library check happens first and is fatal on its own); separately, without `TOPAS_DIR` it also can't auto-resolve `Technical_Reference.pdf` for §21.2 descriptions (pass `--docx`/`--pdf` explicitly instead, or macros are still shown in full, just ungrouped by category).
- `scripts/topas_keyword_tree.py` — has nowhere under `TOPAS_DIR` to default its output to and exits immediately unless `-o` is given explicitly.

If a colleague trying this skill doesn't have `TOPAS_DIR` set, tell them plainly what it unlocks and point them at this section rather than guessing at file contents.

## Scripts

- `scripts/topas_install.py` — resolves file paths against the person's own TOPAS installation via `TOPAS_DIR` (required; this skill bundles no fallback copy — see "Locating your TOPAS installation" above). Used internally by the other two scripts and directly by you when reading an example file:
  ```
  python3 scripts/topas_install.py --inc-dir
  python3 scripts/topas_install.py --example cf/alvo4a.inp
  ```
- `scripts/check_inp_syntax.py` — a basic, heuristic syntax checker. Run it against any `.inp` file or directory before handing it over:
  ```
  python3 scripts/check_inp_syntax.py path/to/file.inp
  python3 scripts/check_inp_syntax.py path/to/directory/    # recurses
  python3 scripts/check_inp_syntax.py path/to/file "macro FILE { file.xy }"   # tc.exe-style extra command-line text
  ```
  To run it automatically against the file currently open in VS Code (rather than typing the command by hand each time), see `references/vscode-integration.md` for a `tasks.json` setup with an optional Problems-panel integration.

  **Extensionless paths and extra command-line text both mirror `tc.exe`'s own real CLI convention (`references/20-miscellaneous.md`, "Batch mode operation"), not a convention invented for this script.** A bare path with no extension (`test_examples/clay`) resolves the same way `tc.exe` does -- tried as-is, then with `.inp`/`.INP` appended (`resolve_inp_path`). A token that follows a file argument and does NOT itself resolve to an existing file/directory is treated as extra text appended to that file's content before checking (`parse_cli_entries` / `check_file`'s `extra_text` parameter) -- exactly matching `tc file.inp "macro FILE { file.xy }"` or `tc file "#define foo"`. This matters for real correctness, not just convenience: `check_symmetry_constraints` and the `#ifdef`-branch stripping are `#define`-aware, so a file that only becomes valid/complete once a particular `#define` flag is injected via the command line (a common convention across the bundled example corpus: a commented-out `'#define create_` at the top, meant to be activated via the CLI rather than uncommented in the file itself) needs that flag passed the same way to be checked against its actually-intended content. Appending at the end (rather than needing to insert at a specific point) is safe because `strip_inactive_ifdef_branches` already does a file-wide `#define` scan before evaluating any branch (not a strict top-to-bottom single pass requiring definition-before-use).

  **A real bug found and fixed while running a large curated batch of real corpus files through this new CLI feature: `#ifdef NAME ... #endif` all on ONE physical line is valid TOPAS syntax (confirmed directly by the user, TOPAS-Academic's author), but `strip_inactive_ifdef_branches` didn't handle it.** The original implementation processed each line with `.search()` against each directive pattern in turn and blanked the ENTIRE line on the first match (`test_examples/1000s-of-patterns/10000.inp` line 140 -- `#ifdef USE_PEAK_BUFFER_SIMILAR_TAG peak_buffer_similar_tag 1 #endif` -- triggered it directly), so a directive-open-and-close sharing one line pushed a stack frame that was never popped (the `#endif` on that same line was never even inspected). Every line after that point was then silently misclassified active/inactive for the rest of the file, feeding `check_braces_and_parens` a text with real `{`/`}` characters incorrectly blanked away -- surfaced as spurious "unmatched opening brace" errors far from the actual (nonexistent) problem. Fixed by processing each line as a *sequence* of directive matches (`LINE_DIRECTIVE_RE.finditer`, a single combined regex with named groups) rather than assuming at most one directive consumes the whole line -- each inter-directive text segment is blanked/kept based on the active state at that exact point in the sequence. Verified: `10000.inp` now correctly reports its one real (and correct) finding instead of three spurious brace errors; a full rerun of the ~170-entry curated batch afterward showed the `FAIL` count drop from 3 to 0 with no new regressions (two of those three "FAIL"s were actually an unrelated path-layout mismatch in the batch list itself, not a real code issue -- see below).

  **Three real exceptions found via this check's own curated-corpus regression sweep, all confirmed as genuine, common TOPAS idioms rather than bugs:** (1) `@` as a bare macro-call argument (`TOF_PV(@, 100, @, .5, t1)`, `CS_L(@, 300)`) -- means "auto-name whatever parameter this call builds here" and is legitimately followed by `,`/`)` with no number in sight, since that's the macro's own business, not a value this checker can see the shape of; (2) `@` directly concatenated with a suggested name and no space (`TOF_PV(@pv6, 72.10546, ...)`, found in a real corpus magnetic-Rietveld file) -- the same concatenated-sigil convention already established for `!name` (e.g. `prm !b1 .5`), just for `@`; the name is consumed first, then the same number-or-`=` rule applies to whatever follows it; (3) a macro body that's just a bare `@`/`@name` and nothing else (`macro SVs { @ }`) -- a common toggle-macro idiom (confirmed directly against a real corpus file, `kaolinite.inp`, using exactly this pattern for several such macros to conditionally enable refinement elsewhere) where `@` is immediately followed by `}`.   **The eleventh check (`check_symmetry_constraints`) validates an existing `.inp`'s ADP and lattice constraints against what its own declared space group actually requires.** Reuses the exact same crystallography engine as `cif_to_str.py` -- both scripts now import it from a shared `scripts/symmetry_utils.py` (extracted from `cif_to_str.py` during this check's development, so the two never drift apart) -- run in the opposite direction: instead of generating correct syntax from a CIF, it parses the `.inp`'s own already-written text and compares it against what `classify_adps`/`classify_crystal_system` say the space group requires. Space-group operators are resolved via `symmetry_utils.resolve_sg_operators` (TOPAS's own `sgcom6.exe`/`sg/` database -- an `.inp` file never carries its own operator list the way a CIF can, so this is the *only* source here, not a fallback); needs `TOPAS_DIR`, and silently produces no findings (not an error) if it's unavailable, the same degrade-gracefully pattern as the macro-arity check's live-install fallback. Built-in lattice macros (`Cubic`/`Tetragonal`/`Hexagonal`/`Trigonal`/`Rhombohedral`, confirmed by reading their actual `topas.inc` bodies -- e.g. `Rhombohedral(a_cv, al_cv)` ties `b = Get(a); c = Get(a); be = Get(al); ga = Get(al);` internally) are recognized and skipped for the literal `a`/`b`/`c`/`al`/`be`/`ga` checks, but cross-checked against the space group's own computed crystal system -- e.g. `Cubic(...)` called on a space group that actually resolves to tetragonal is flagged, since that macro would wrongly force `a=b=c`.

  **Site COORDINATE (x/y/z) constraint checking was built, then later REMOVED entirely -- both directly at the user's own request (TOPAS-Academic's author), a real reversal worth understanding rather than just a historical footnote.** It originally flagged a coordinate refined independently (bare or `@`) when site symmetry would otherwise tie/fix it (e.g. `y` written independently when the position requires `y = Get(x);`). The user later hit this on a real file (`test_examples/temp.inp`'s `O1`/`Ti1` sites) and corrected the premise directly: *"the act of refining on x or y independently means that TOPAS will treat these coordinates as being different to each other"* -- i.e. writing a coordinate independently (a bare/named parameter with no `!`) is not an accidental omission that risks drifting off a special position, it IS the deliberate signal that the site should be treated as a general position, full stop, regardless of what numeric value it currently happens to hold. Confirmed with a precise minimal-pair example: `site Ti1 x xti1 0 y yti1 0 z zti1 0 ...` (all three named, no `!`) is a **general position** even though the written values are `(0,0,0)` -- refinement is free to move them anywhere, independently; `site Ti1 x !xti1 0 y !yti1 0 z !zti1 0 ...` (all `!`-fixed) is the only form that's actually a **special position** in a way TOPAS enforces, since those coordinates genuinely cannot move. The whole `for coord in ("x","y","z")` block (both the "fixed" and "tied" cases) was removed from `check_symmetry_constraints` as a result -- `classify_coordinates` (still in `symmetry_utils.py`, still used by `cif_to_str.py`'s forward CIF-to-`str` direction) is simply no longer called here. **This does not extend to the ADP or lattice checks below** -- neither has been challenged the same way, and the ADP check in particular was already a fundamentally different kind of check (see below) rather than a smaller version of the same mistake. Re-verified after removal against the full curated `tcinps-2.bat` corpus (162 present files): 0 exceptions, 0 coordinate-tie/fixed warnings remaining, ADP and lattice warnings both confirmed still firing correctly (unaffected).

  Design points worth knowing before trusting a hit (or a clean pass) from this check:
  - **ADP validation (`u11`..`u23`) was never tautological the way the now-removed coordinate check was, and catches genuine static numeric errors regardless of refinement status.** A coordinate's "required" fixed value came from that same coordinate's own written number, so a bare value could never numerically mismatch it -- refining it away (`@`) was the only real risk, and that's exactly the class of warning the user determined was unwanted, hence its removal above. An ADP's required value comes from `classify_adps(stabilizer)`, derived purely from the site's *position* -- entirely independent of whatever `u_ij` number happens to be written. So `'u12 0.02'` when site symmetry requires it fixed at exactly 0, or a bare value that doesn't match the numeric target of a required tie (e.g. `u13` should equal `u12`'s value but doesn't), is a real, always-flagged data error -- not gated behind "is anything being refined," and not affected by the coordinate-check removal above. Verified with two deliberate-bug test files: one on ZrW2O8's Pa-3 3-fold-axis site symmetry (catches an independently-`@`-refined `u22` that should track `u11`, and a bare `u13` with the wrong numeric value), one on Fm-3m's fixed-to-zero off-diagonal terms (catches an `@`-refined `u12` that must be exactly 0, and a bare nonzero `u13`) -- both confirmed to produce zero false positives on the already-`tc.exe`-verified correct files. Also confirmed directly on `temp.inp`'s rutile `P4_2/mnm` site: nonzero `u13`/`u23` correctly still flagged as symmetry violations even after the coordinate check was removed.
  - **A real bug found and fixed during this work: `resolve_site_coordinates` originally only recognized `Get()`-based equations, missing a plain-constant equation like `x = 1/4;`** -- exactly the form `cif_to_str.py` itself emits for a fixed coordinate that snaps to a common fraction. `resolve_site_coordinates` itself (and `parse_plain_numeric_equation()`) remain in use -- by `plot_str_3d.py` and by the ADP check here (which still needs each site's concrete point resolved first, even though the point's own tie/fixed status is no longer separately flagged) -- so this fix is still load-bearing, only the coordinate-tie WARNING built on top of it was removed.
  - **`find_str_blocks` didn't terminate a str's span at a `for` keyword, so a `for xdds { for strs {...} for strs N to M { space_group ... } }` loop appearing later in the file could get silently absorbed into an earlier, unrelated bare str's span.** Real bug, found and diagnosed directly by the user (TOPAS-Academic's author) on a real file (`test_examples/matthew-rowles/2457_surface_paper_2.inp`): several bare `str`-only "reporting" objects (auxiliary density/weight-percent computations, no `space_group` of their own) appear well before the file's actual phase-defining structure, which is set up via `for strs 1 to 2 { space_group P_42/M_N_M ... }`, `for strs 3 to 4 { space_group P1 ... }`, `for strs 5 to 6 { space_group P-1 ... }` -- `for strs N to M` applies keywords to a RANGE of already-declared str objects by their 1-based sequential index, a legitimate TOPAS idiom entirely separate from writing `space_group` inline inside a literal `str` block. Since `for xdds`/`for strs` opens with `{`, the old span-walker (which only stopped at another literal `str` keyword or its own depth reaching 0) walked straight through the whole loop region, so one of the earlier bare strs' span silently swallowed all three `space_group` declarations -- `check_symmetry_constraints` then attributed the FIRST one found (`P_42/M_N_M`, via a plain `re.search`) to unrelated content via a totally different phase, producing a nonexistent "lattice length must be `Get()`-tied" warning. Fixed by also terminating a str's span at a `for` keyword occurring at the same relative brace depth (mirroring the existing "another `str` keyword" termination). Re-verified: the false positive is gone with no new symmetry-constraint findings anywhere in that file, and a full rerun of the ~170-entry curated batch showed byte-identical WARN/FAIL file sets before and after the fix (only that one file's internal issue list changed). Known remaining gap: a str's OWN literal `space_group` (written directly inside its bare block, the common case throughout the rest of this skill's corpus) is unaffected by this fix either way; only the `for strs N to M`-assigned-space-group idiom was ever at risk, and that idiom is now correctly left unresolved (silently skipped, not guessed at) for str objects it doesn't literally touch.
  - **(Historical, applied to the now-removed coordinate check) This checker was numerically-, not symbolically-, driven** (same as `cif_to_str.py`): it derived "what does site symmetry require" from the coordinates' own resolved values. A badly-wrong tie equation (the right target, wrong offset) could shift the resolved point enough that it no longer sat on the special position at all -- confirmed with a dedicated test (`y = Get(x) + 0.1;` where the true tie has zero offset produced no warning, because the resulting point genuinely isn't on the 3-fold axis anymore). Recorded here for context since the same numerically-driven approach still underlies the ADP check above, even though the coordinate-tie warning itself is gone.
  - **(Historical, applied to the now-removed coordinate check) The 'tied' coordinate check only ever accepted the literal `Get(other)+offset` syntax as "matching" its derived requirement -- rejecting a coordinate written any other way, even when that other way is numerically (and structurally) just as safe.** Real bug, found and diagnosed directly by the user (TOPAS-Academic's author) on two real files (`test_examples/clay.inp`, `test_examples/k-factor/k-factor.inp`): both wrote a tied coordinate pair as independent PLAIN-CONSTANT equations (e.g. clay.inp's `x =0; y =0;`; k-factor.inp's `x =1/3; y =2/3;`, numerically consistent with a required `y = -Get(x);` since 2/3 = -1/3 mod 1) rather than the literal Get()-tie form -- flagged as a mismatch anyway at the time. Also recorded here for context (`extract_keyword_form`'s `'value'` tuple carrying the given parameter name, 5th element, is what this fix added, and it's still used elsewhere) even though the warning built on top of it no longer exists.

  **`GET_TIE_RE`/`resolve_site_coordinates` handle a `Get()` tie with an optional multiplier, not just an optional additive offset -- and that multiplier can be IMPLICIT (bare juxtaposition, no `*`), which is real, valid TOPAS syntax, not a typo.** Found via a direct test: the user hand-edited `test_examples/simple.inp`'s O2 site to `z = Get(x) .99 + 0.1625;` and asked whether TOPAS equations are evaluated in general -- running that exact file through a live `tc.exe` confirmed it parses and refines with zero syntax error, meaning `Get(x) .99` really does mean `Get(x) * 0.99` by plain adjacency. The original regex only recognized `Get(other)` plus an optional `[+-]offset`, so it silently failed to resolve O2's coordinates at all (both here and in `plot_str_3d.py`, which shares this exact code -- O2 was dropped from the 3D render entirely, no error, just quietly absent). `GET_TIE_RE` now has named groups for an optional multiplier (`* 0.99` or bare `0.99`) in addition to the offset, with a shared `get_tie_value(match, known_value)` helper so all three call sites at the time (`resolve_site_coordinates`, the lattice-length-tie check, and the Wyckoff-position-tie check) computed it identically -- the Wyckoff-position-tie check (the site x/y/z coordinate tie warning) was later removed entirely (see "The eleventh check" above); `resolve_site_coordinates` and the lattice-length-tie check are still active and still share this same mechanism. The "exact tie required" lattice check (a lattice length must equal `Get(a);` with nothing else) was updated to explicitly require *no* multiplier for a match, not just no offset -- a scaled tie is a real mismatch there, not something this relaxation should start silently accepting. Re-verified against the full `test_examples/` corpus (271 files, zero crashes) plus a direct spot-check on a known lattice-tie file (ZrW2O8, still passes clean) after the change. Still explicitly NOT a general equation evaluator: multi-term sums of two different `Get()`s, nested functions, trig/built-ins, and references to an external `prm` are still unresolved and skipped with a note, not guessed at.

  **`GET_TIE_RE` also supports an explicit divisor (`Get(a) / 100`), and `resolve_site_coordinates`/`plot_str_3d.py` now understand `Get()`'s real SCOPE-WALKING behavior, empirically confirmed against a live `tc.exe` rather than assumed.** The user described TOPAS's internal object model as a tree, where `Get()` searches the current scope first and walks up if not found there -- true, but with a crucial, non-obvious refinement pinned down by changing exactly one thing at a time in a known-working file (`test_examples/simple.inp`, via a scratch copy) and rerunning `tc.exe` after each change:
  - `beq = Get(b1);` (`b1` a str-level `prm !b1 .5`, the SAME name that already works fine as a BARE reference, `beq = b1;`) **FAILS**: `Cannot locate b1 from beq in data structures`. So `Get()` is NOT a general named-variable lookup -- a plain `prm`/`local` is referenced by ordinary bare identifier, using its own separate (and apparently more permissive) scoping; it is never wrapped in `Get()`.
  - `beq = Get(a) / 10;` (the site itself has no `a` of its own; the enclosing `str`'s `a` here comes only from `Cubic(@ 10.820412)`, never written as a literal `a` token) **SUCCEEDS** and refines correctly. So `Get()` walks up to the NEAREST ENCLOSING object that owns a keyword actually named `a` -- a real structural slot (`x`/`y`/`z`/`u11`..`u23`/`occ`/`beq` for a site; `a`/`b`/`c`/`al`/`be`/`ga`/`scale`/`space_group` for a `str`; ...), not just the current object, and not arbitrary user-declared names.
  - This matches (and explains) the manual's own `fn lat(h,k,l) = h Get(a) + k Get(b) + l Get(c);` example (`references/20-miscellaneous.md`) -- when inlined into a `str`'s own equation, `Get(a)` reads THAT str's own lattice `a`, the nearest enclosing object that has one.
  - New shared function `resolve_str_scope_values(preamble)` in `check_inp_syntax.py` resolves a `str` block's own `a`/`b`/`c`/`al`/`be`/`ga`/`scale` (handling both literal keywords and the `Cubic`/`Tetragonal`/`Hexagonal`/`Trigonal`/`Rhombohedral` macros, argument mapping read from `topas.inc` the same way `plot_str_3d.py`'s `extract_cell_params` already did) -- with NO invented defaults, since a `str` genuinely missing `b` really does make `Get(b)` fail in real TOPAS too, not silently succeed. `resolve_site_coordinates` now takes an optional `outer_known` dict (this str-scope dict) and falls back to it for any `Get()` reference not among the site's own `x`/`y`/`z` -- mirroring the confirmed search order (current object, then nearest ancestor that owns the keyword). Both call sites (`check_symmetry_constraints`, `plot_str_3d.py`'s `gather_atoms`) now pass it. Verified end to end: a synthetic site equation `z = Get(a) / 100 + 0.0125;` (previously silently skipped -- no error, just absent from the 3D render) now resolves correctly in both scripts, confirmed against the same `tc.exe`-computed value.
  - Division support in `GET_TIE_RE` was found missing as a direct side effect of this same test (`Get(a) / 100` didn't match at all, only the multiplication form did) -- added symmetrically to the existing multiplication/juxtaposition support (`mul_op` now holds `*` or `/`; there is no implicit-juxtaposition form for division, only `*` has that).
  - **Practical implication for future `.inp` work**: don't try to make `Get()` "reach" a plain `prm`/`local` name -- that's not a gap in this skill's tooling, it's a fact about TOPAS itself (confirmed by the exact same error a real refinement run produces). Only wrap a reference in `Get()` when referencing an actual structural keyword slot on the current object or an enclosing one; reference a `prm`/`local` by bare name instead.

  **The tenth check (`check_bom`) is a hard error, not a heuristic warning: a leading BOM makes TOPAS choke on the file's very first token every single time, with no legitimate exception.** Confirmed as a real, recurring problem directly by the user (TOPAS-Academic's author) -- `test_examples/simple.inp` picked up a BOM at least twice during this skill's development, breaking `tc.exe` with a cryptic `unknown or misplaced keyword` at LINE 1 despite the file looking completely normal in a text editor (most editors render the BOM as invisible, so there's no visual clue anything is wrong). Apparently reintroduced by an external editor/tool between edits, not by anything this skill's own scripts do (they write plain UTF-8 without a BOM throughout -- every scratch/conversion script in this skill uses `New-Object System.Text.UTF8Encoding $false` or Python's default `open(..., encoding="utf-8")`, neither of which emits a BOM). Checking for it here means it's caught automatically before a run, rather than relying on remembering to check by hand each time it recurs.

  **The ninth check (`check_xyz_near_one_third`) was raised directly by the user (TOPAS-Academic's author): a common CIF-import artifact is a site sitting exactly on a 3-fold axis (symmetry-fixed to 1/3 or 2/3) getting written out by the CIF as a rounded decimal and pasted into the INP as a bare free-refined value instead of the exact equation form -- free-refining a coordinate the space group actually fixes at an exact special-position value is a real crystallographic bug, since it can drift away from the symmetry-required exact fraction during refinement.** Scoped to the bare `x [!@]NAME? VALUE` forms (same value-parsing as the eighth check); an equation form (`x = ...;`) is skipped since a deliberate equation was presumably already written. Values are reduced mod 1 before comparing (tolerance `0.0015`, catching CIF-rounded decimals from 3 to 5 places), so a translated coordinate like `-0.6667` (fractionally equivalent to `0.3333`) is still caught. Flagged as a **warning**: corpus-verified against the full `test_examples/` corpus, 4,472 occurrences across 62 files -- but ~97% concentrated in just 6 large synthetic/simulated multi-atom cells (`je-pdf/pdf_662_01.inp`, `je-pdf/bragg.inp`, `pdf/philip/Phils-talk/04 SrTiO3_5K_largebox_START.inp`, and similar -- 1,400+ hits each) where near-1/3 coordinates are common and usually intentional (grid/supercell placement), not CIF-rounding artifacts. **Treat a large hit count on a big multi-atom/simulated-box file as expected noise, not evidence of that many real bugs** -- this check is far more diagnostic on a normal, modest-atom-count, CIF-derived structure, which is the case it was written for.

  **The eighth check (`check_single_e_arg_keywords`) generalizes the fifth check's `$`-sigil approach to `E`-sigil (numeric-value) keywords.** `load_single_e_arg_keywords()` harvests every keyword documented in bracket notation with exactly one `E`/`!E` argument and nothing else (e.g. `[scale E]`, `[beq E]` -- 212 keywords as of this writing before filtering), using the same conflict-detection as the `$`-arg harvester: a keyword documented anywhere with a different shape is dropped entirely. Because `E` covers several valid written forms (bare value, `@`-auto-named, `!`-fixed, explicitly named, or a full `= ...;` equation, all skipped here since they terminate at `;`, not this check's concern), the check consumes at most one leading sigil/name token before matching the numeric value itself, then flags only when the very next token is *itself* another bare number with no recognized attribute keyword (`min`/`max`/`del`/...) or new statement in between -- there's no legitimate TOPAS grammar where two bare numbers can follow a single-`E`-arg keyword's value back to back. Restricted to keywords of length `>= MIN_E_ARG_CHECK_LEN` (4) as a deliberate safety margin against 1-3 character names (`a`, `b`, `c`, `x`, `y`, `z`, `al`, `ta`, `u11`, `min`, `max`, `del`, `occ`, `beq`, ...) that are common enough as ordinary equation variable names or chained attribute keywords to carry real false-positive risk even though none tripped it empirically -- mirrors `MIN_TYPO_CHECK_LEN`'s same rationale for the keyword-typo check. Verified against the full `test_examples/` corpus (1,121 `.inp` files): **zero false positives**, and the one real hit was the genuine `scale ... 907` bug above. Given the corpus can't prove every one of the ~176 surviving harvested keywords' real-world grammar is fully captured by its bracket notation (the `$`-arg check needed hand exclusions for `site`/`hkl_plane` for exactly this reason, found only empirically -- see the fifth-check history below), this is flagged as a **warning**, not a hard error, the same caution applied to the fifth check.

  **Both the fifth and eighth checks encode the same general grammar rule, confirmed directly by the user (TOPAS-Academic's author): once a keyword's value/argument(s) are consumed, the very next token must be a preprocessor directive (`#define`, `#ifdef`, `#include`, ...), a recognized keyword, or a macro name** (macros can themselves expand to further macros/keywords, but the check only needs to confirm the *immediate* next token is *a* macro name -- TOPAS's own preprocessor handles the recursive expansion afterward, so this checker never needs to trace it). The keyword/macro-name half of that rule was already covered by both checks' shared `keywords_plus_macros` lookup set. The preprocessor-directive half was not, initially: neither check's "next token" logic explicitly recognized a leading `#` as valid -- it worked only *incidentally*, because each check's final flagging filter happened to be narrow enough (the `$`-arg check only flags letters/digits/colon/slash/hyphen-shaped fragments; the `E`-arg check only flags fragments that are themselves an exact number) that a `#`-led token never matched either filter and so was silently never flagged, not because it was recognized as legitimate. Confirmed via a dedicated test (a single-arg keyword's value immediately followed by `#define FOO` on the next line, for both checks) before and after: zero warnings either way, so this was never a live false positive -- but relying on a filter's incidental narrowness rather than an explicit rule is fragile (a future broadening of either filter's "plausible fragment" shape would silently reintroduce it). Fixed by adding an explicit `if next_token[0] == "#": continue` immediately alongside each check's existing structural-character skip (`{}();,`), making the rule intentional rather than emergent. Re-verified against the full `test_examples/` corpus afterward: still zero false positives.

  **The seventh check (`check_prm_local_missing_min_max`) is a best-practice warning, not a syntax check, raised directly by the user (TOPAS-Academic's author).** Unlike subject-specific keywords (`beq`, `CS_L`, lattice parameters, ...), which all carry internal default `min`/`max` limits (Table 2-1 in `references/01-syntax-and-parameters.md`), `prm`/`local` have none at all -- a refined parameter (no `!` prefix) declared with a bare value and no `min`/`max` can wander to a numerically invalid or unphysical value during refinement with nothing to stop it (see that same reference file, "Try and use parameter attributes"). Deliberately scoped to only the plain `prm|local [!@$]NAME VALUE` form: `!`-prefixed names are skipped (not refined, so moot), the anonymous auto-name form (`prm @ 0.5`, no identifier at all) is skipped (nothing to name in the warning), and equation forms (`prm name = ...;`) are skipped entirely, since per section 2.9 an equation is only an independently refined parameter if it happens to reduce to a constant -- undecidable from static text, so not attempted. Corpus verification against the full 280+ example set turned up a real false-positive class before this was trusted: `test_examples/1000s-of-patterns/fit.inp` uses a common project-local shorthand, `macro MM(x1, x2) { min = x1; max = x2; ...}`, called right after the bare value (`prm cs_g1_ 30 MM(3, 1000)`) to set min/max indirectly -- a literal-token search for `min`/`max` alone misses this entirely. Fixed by `harvest_macros_with_min_max_body()`, which scans the *current file's own* macro definitions (not the shared `.inc` library) for any whose body contains a literal `min`/`max`, and treats a call to one of those macros within the search window as equally satisfying the check. The forward search for `min`/`max` (or a qualifying macro call) is bounded by a small, curated set of real statement-starting keywords (`PRM_LOCAL_BOUNDARY_KEYWORDS`) rather than the full keyword/macro list, specifically to avoid a capitalized built-in `Min(...)`/`Max(...)` **equation function** call (e.g. inside an `update`/`val_on_continue` clause) being misread as a new statement and truncating the scan before a legitimate later `min`/`max` attribute is reached -- confirmed with a dedicated test case before trusting it.

  **The sixth check (`check_missing_colon_before_value`) is the mirror image of the semicolon check.** TOPAS's value-report convention is `<equation>; : <value>` (see `references/01-syntax-and-parameters.md`, "Reporting on equation values") -- the semicolon check above catches a dropped `;` before a `:`; this one catches the opposite mistake, a dropped `:` after a valid `;`, which leaves the value as a bare orphaned number with nothing referencing it (not part of any equation, not a keyword's argument). Unlike the single-string-arg check, this one is a hard **error**, not a warning: there's no legitimate TOPAS grammar where a bare number validly follows `;` with nothing between them, so there was no keyword-specific false-positive risk to reckon with here the way there was for `site`/`hkl_plane` above -- verified against the full 286-file example corpus with zero false positives regardless.

  **The fifth check is driven directly by Technical_Reference.pdf's own Chapter 2 notation, not a guess.** The user confirmed the bracket-notation sigil convention used throughout the manual: `$name`/`$symbol`/`$file` means a bare token or one quoted string; `#value` means a plain number; `E` (often written `!E`) means an equation-or-value parameter slot (bare value, `@`-auto-named, `!`-fixed, explicitly named, or a full `= ...;` equation). `load_single_string_arg_keywords()` in `check_inp_syntax.py` harvests every keyword documented with a bracket signature of *exactly one* `$`-sigil argument and nothing else (51 keywords as of this writing: `phase_name`, `layer`, `stack`, `site` -- see exclusion below --, `hkl_plane`, `cloud`, `pick_atoms`, `out_rwp`, and more) and applies the same stray-space check to all of them generically, rather than the space_group-only special case this started as.

  Generalizing this safely took real iteration, not just writing the harvester once -- each round found something the previous one missed, and each was verified by re-running the full 280+ example corpus, exactly like the semicolon-check fix below:
  - **Widened the trusted keyword set.** `macro`, `prm`, `local`, `for`, `inp_text`, `fn`, and `load` are real low-level TOPAS keywords the bracket-notation harvest doesn't capture (they're language constructs, not `[keyword ...]`-documented entries) -- without them, a real file with e.g. a `macro` definition right after a `space_group` line produced false positives.
  - **`site` and `hkl_plane` needed hard exclusion** (`SINGLE_ARG_FALSE_POSITIVES` in the script) despite having a clean single-`$`-arg bracket entry: `site`'s `[site $site]` entry only documents its narrow use inside a stacking-faults `layer { }` block, never its everyday multi-argument form (`site Zr x 0 y 0 z 0 occ Zr+4 1 beq b1 0.5`), which is only ever shown in prose/worked examples, not its own bracket line. `hkl_plane`'s `$hkl` sigil is a conceptual label for "an hkl index" but real usage (`hkl_plane 1 1 1`) takes three separate numbers, not one token. Neither conflict was visible to the harvester itself (no *other* bracket entry documented the richer form to trigger the automatic conflict-detection below) -- both were found only by running against real files and reading what actually broke.
  - **Two real bugs in the check's own regex, not the grammar model.** `[ \t]*(\S+)` (zero-or-more whitespace) let a keyword *referenced* inside another expression with no space at all -- e.g. `Out(Get(phase_name), "...")`, where `phase_name` is immediately followed by `)` -- get misread as if `)` were its value; fixed by requiring `[ \t]+` (at least one real space), since a genuine value-taking use is always keyword-SPACE-value. Separately, `layer {` (opening a block, a second legitimate grammar for `layer` not captured by its bracket entry) had its `{` misread as a value token; fixed by skipping outright whenever the first captured "value" is itself a structural character (`{}();,`).
  - **The harvester itself had a subtler bug** in what counts as a "conflicting" bracket entry: `[space_group $]` (a bare `$` with no name after it, in `22-charge-flipping.md`'s condensed summary table) was being treated as a *different* shape than `$symbol` elsewhere, which caused `space_group` -- the keyword this whole feature was built to catch -- to get silently dropped from the harvested set entirely. Fixed by treating a lone `$` as the same single-string-argument evidence as `$word`, not a conflict.

  This history is worth reading before extending this check further: the harvester's automatic conflict-detection (dropping a keyword if it's *ever* documented with a different shape) only catches conflicts the manual documents in more than one place: it did nothing for `site`/`hkl_plane`, whose richer real grammar isn't captured in bracket notation anywhere. Trust a clean-looking harvest only as far as the corpus re-verification actually confirms it, not further.

  **The keyword-typo check (item 3) was substantially rewritten, at the explicit, direct instruction of the user (TOPAS-Academic's author): "There should be no minimum length for typos, MIN_TYPO_CHECK_LEN should be set to zero. An identifier must be a keyword or a macro. When a file is macro expanded then the identifier must be a keyword."** The trigger was a real miss: a stray, meaningless bare word (`eee`) sitting alone on its own line in `test_examples/simple.inp` passed cleanly, for two independent reasons -- it was under the old 5-character length floor, AND it didn't closely resemble any real keyword (the old check only ever flagged a *near-miss* of a *specific* known name; a token with no resemblance to anything was silently allowed through even without the length floor). Fixing this properly meant more than deleting the length constant: the checker also had to learn the actual argument grammar of every keyword/macro that consumes more than the single next token (**confirmed directly by the user as the guiding principle**: "It knows that z_matrix will load a string as the next sequence on the stream. Therefore you already know the syntax of z_matrix (or any other keyword) and therefore if [a stray token] occurs outside of any keywords then that must be an error") -- otherwise the newly-strict check would have flagged enormous amounts of entirely legitimate syntax as false positives. A full curated-corpus regression sweep (`tcinps-2.bat`'s 197 files) went from **901 false-positive warnings across 147 files, down to 0 real false positives** (7 residual warnings across 2 files are genuine: locally-`#include`d macros this checker doesn't resolve, honestly hedged as "a gap in this checker, not a mistake in the file"; and one real, unfixed finding in a real corpus file, `extra_X`, which the manual only documents as `extra_X_left`/`extra_X_right`, never bare) across roughly fifteen rounds of fix-then-reverify. The fixes, in the order they were needed:
  - **The keyword harvester itself (`load_keyword_set`) only ever captured the FIRST name inside a multi-name bracket** (`\[(\w+)` matched just `pv_lor` out of `[pv_lor E  pv_fwhm E]`, silently losing `pv_fwhm`, `h2`, `m1`, `m2`, and everything past the first name in any bracket packing several keywords together) -- rewritten to bracket-match properly and extract every keyword-shaped token inside, skipping only recognized type-marker placeholders (`E`, `#`, `$name`, `...`) via `extract_bracket_keyword_names()`/`BRACKET_TYPE_MARKER_RE`.
  - **`MACRO_DEF_RE`/`BODILESS_MACRO_DEF_RE` didn't accept the optional `&` before a macro name** (`macro & CeV(c, v)`, the same construct `expand_inp_macros.py` was fixed for earlier) -- `topas.inc`'s own `CeV` macro, used constantly throughout the library, was invisible to this checker's own harvest as a result.
  - **A handful of equation functions/keywords are documented as a bare signature line with no brackets or table row at all** (`Constant(expression)`, `Bkg_at(x)`, `lpsd_equitorial_sample_length_mm E`) -- added `BARE_FUNC_SIGNATURE_RE`/`BARE_KEYWORD_SIGNATURE_RE` to catch these two remaining documentation shapes.
  - **`z_matrix`'s own atom-label/reference-label tokens** (`C2  C1  =dc1c2;`), **`space_group`'s symbol** (hyphens/digits inside it, e.g. `R_-3_c`, fragment `IDENTIFIER_RE` into multiple pieces), **a keyword/macro call's own parenthesized argument list**, and **a small curated set of keyword-prefixed brace-block "data tables"** (`load site x y z occ beq layer { ... }`, `ADPs { u11C1 0.01 ... }`, `#list File_Name Time Temperature Gas { ... }`) all needed dedicated span-finders (`find_z_matrix_label_spans`, `find_space_group_symbol_spans`, `find_call_argument_spans`, `find_data_block_spans`) to exempt their own internal bare tokens from being checked as if they were independent keyword references -- deliberately a small, curated trigger-word list (`DATA_BLOCK_TRIGGER_WORDS`), not "any keyword followed by `{`", since a macro's own body (`macro Foo(x) { ...real code... }`) has the identical surface shape but must still be checked normally.
  - **A complex bare (unquoted, hyphenated) filename breaks into several separate identifier fragments** (`cube-ln-normal-1.xy` tokenizes as `cube`/`ln`/`normal`/`1`/`xy`; `CT-DMF(5oC)-100K-6-64-VC.raw` similarly) -- `find_filename_run_spans`/`FILENAME_RUN_RE` matches the whole hyphen-or-paren-containing run ending in a recognized extension as one span.
  - **Macro/`fn` PARAMETER names were never registered as legitimate identifiers throughout their own body** -- only each macro/fn's own NAME was harvested, never its argument list, so e.g. spherical-harmonics-style macro-internal names (`sh_c22m`, `y22p`, `Lpa`, `u1`, `v1`) or a `fn`'s own math variables (`x1`, `y1`, `z1`, `o1`, `b1`) were flagged as unrecognized throughout the very body that declares and uses them. Fixed in `find_defined_names()` by harvesting `MACRO_DEF_RE`/`FN_DEF_RE` parameter lists directly. Relatedly, `find_call_argument_spans` checked a callee name against `keywords_plus_macros`, which never included `fn`-defined names (only `macro`-defined ones) -- added `harvest_fn_names()` so a call like `site_F(h, k, l, x1, y1, z1, o1, b1, ...)` to a real `fn site_F(...)` gets its whole argument list exempted the same way a macro call's does.
  - **A reordering bug in the main token-scan loop caused a cascading failure across any dense, multi-keyword line.** The original loop checked `if idx == skip_next_idx` (the leftover-skip-window test) *before* `if word in keywords`. A keyword whose own argument produces no `IDENTIFIER_RE` token at all (a bare number, e.g. `x 0.25`) leaves the skip window pointing at the *next* keyword in the token list (`y` in `x 0.25 y 0.5`) purely by coincidence of token-list adjacency -- checked in the wrong order, that next keyword gets silently swallowed as if it were a skipped argument, and the whole skip chain desyncs for the rest of the statement (an entire `site NAME x # y # z # occ ATOM # beq #` line, in the worst case). Fixed by checking `word in keywords` first, unconditionally, so a real keyword is always recognized regardless of stale skip state.
  - **A keyword/macro invoked as a parenthesized call left its skip window "dangling" onto whatever came after the call closes.** `ZE(@, 0)` immediately followed by a leftover `eee` on the next line -- since neither `@` nor `0` are identifier-shaped, the skip window set for `ZE` pointed straight at `eee`, silently absorbing it as if it were `ZE`'s own bare tag-name argument. Fixed by only setting the skip window when the keyword is NOT immediately followed by `(`.
  - **`occ`/`element_weight_percent` take TWO bare tag tokens before the real value, not one** (`occ $atom [$name] E`, `element_weight_percent $atom $Name #`) -- `WIDE_TAG_KEYWORDS` widens the skip window specifically for these.
  - **A tag name declared via a `keyword $name value` convention (`beq`, `occ`, `number_of_sequences`, `element_weight_percent`, ...) was only ever protected positionally, at its own declaration site -- not registered as a persistently known name for later references elsewhere in the file**, and a reference can legitimately appear *before* its own declaration (`number_of_sequences Nseqs 200` declares `Nseqs`; an earlier line's `scale = s 1e-6 / (Nv Nseqs);` references it first). Fixed by moving this registration into a dedicated whole-file pre-pass in `find_defined_names()` (not a side effect of the main flagging loop, which only ever sees things in forward order).
  - **`def NAME1, NAME2;` (a comma-separated multi-name forward declaration with no value at all) only ever had its first name protected** by the general single-token skip -- extended `find_defined_names()` to capture every comma-separated name in this specific form.
  - **TOPAS's `##` token-pasting operator** (`prm ZZ##z1`, building a dynamic name from a macro-argument prefix and a literal hand-typed suffix) **left the suffix half exposed** -- any identifier immediately preceded by `##` is now registered as a legitimate paste-suffix, never independently checked.
  - **Backtick-error-value fractional digits, and TOPAS's own `_LIMIT_MIN_`/`_LIMIT_MAX_` refinement-limit suffix, tokenize as bare identifiers purely because `IDENTIFIER_RE` allows a leading underscore** (`52.5347835\`_0.817...` produces a stray `_0` token, matched up to the decimal point; `...4.81474_` with the error digits apparently truncated produces a bare trailing `_`) -- both recognized and exempted by position (immediately preceded by a backtick or a digit).
  - **A string-literal placeholder (`strip_comments_and_strings` replaces a quoted string's interior with repeated `x` characters, not blanks, so macro-arity checking still sees non-empty content) reads as its own fake identifier** (`space_group "FM3M"` produces a literal `xxxx` token) -- exempted via a `^x+$` check.
  - **The extension half of a bare filename immediately following a keyword wasn't exempted, only the stem half was** (`foo.xy` -- the existing filename-suppression only protected `foo` by checking what follows it for `.ext`; the separate `xy` token right after the literal dot had nothing protecting it) -- added a symmetric check for a token immediately preceded by `.`.
  - Small, hand-verified additions for real names/keywords documented in a shape no harvester above catches at all: `PREPROCESSOR_WORDS` (`#define`/`#ifdef`/`#if`/`#else`/`#elseif`/`#endif`/`#include`/`#list`/`#undef`/`#seed`/macro-internal `#m_if`/`#m_else`/`#m_elseif`/`#m_endif`/`#m_ifarg`/`#m_prm`/`#m_argu`/`#delete_macros` -- the leading `#` isn't a word character, so it strips away during tokenization, leaving the bare directive word looking like an unrecognized identifier), `AXIAL_CONV_SUBPARAMS` (`axial_conv`'s own `filament_length`/`sample_length`/`receiving_slit_length`/`primary_soller_angle`/`secondary_soller_angle`/`axial_n_beta`), `SUPPLEMENTAL_MISC_KEYWORDS` (`atomic_interaction`'s `ai_sites_1`/`ai_sites_2`/etc., documented as a single run-on prose line; `axial_del`, `INP_File`, `Get_Element_Weight`, `fft_max_order`), and `load_reserved_parameter_names()` (harvests `references/01-syntax-and-parameters.md`'s "Table 2-2. Reserved parameter names."/"Table 2-4. Phase intensity reserved parameter names." -- `X`, `Xo`, `Yobs`, `Ycalc`, `H`, `K`, `L`, `D_spacing`, `Th`, `Lam`, `T`, `Cycle`, `Val`, `Change`, and more -- these are internally-updated names usable inside equations, never declared via `prm`/`local` and never a bracket-notation keyword either, so they were a large, real false-positive class on their own before this).

  The known-name set for the keyword-typo check itself is assembled from three sources to keep exact-case matching from misfiring: bracket-notation keywords harvested from *every* reference chapter (not just `21-keyword-index.md` -- some keywords, like rigid-body's `point_for_site`/`translate`, are only bracket-documented in their own topic chapter), every macro name (both `macro Name(args) {}` and bodiless `macro Name {}` forms) from the `.inc` library and the file itself, and a small hand-verified supplemental list for real names that fall through both harvests (documentation-table gaps, or manual-prose case inconsistencies resolved by treating real working examples as ground truth -- see the `SUPPLEMENTAL_KEYWORDS` comments in the script for specifics). It is NOT a full TOPAS parser — it doesn't understand macro expansion, and the keyword-typo check only flags near-misses of a known keyword (an entirely invented/unrecognized keyword that doesn't resemble anything real, like a version-specific or removed keyword, won't be caught — see `references/console-output-and-errors.md` for real examples of that class of error). The arity check has one known blind spot, itself discovered through this project: when a file-local macro reuses a name already defined with a different arity by a system `#include` (as happens with `Beq` in `test_examples/pdf/beq-2-create.inp`, colliding with `pdf.inc`'s unrelated 8-argument `Beq`), TOPAS can fail to resolve the file-local overload at runtime even though the argument count matches a known definition — a clean pass from this checker means "matches a documented arity," not an ironclad guarantee. Verified during development against 280 real example files with zero false "FAIL"s and a single residual false-positive warning after tuning; still, treat every "warning" as a prompt to check manually, not as a confirmed bug. Exit code is 1 only for brace/paren/semicolon errors (hard problems); keyword-typo and arity warnings never fail the exit code.

  The semicolon-termination check itself had a real blind spot, found and fixed after it missed a genuine bug: it originally scanned forward from an `=` for the *next* top-level `;` regardless of which statement that `;` actually belonged to, so a missing semicolon could be silently masked whenever a *later*, unrelated equation in the same scope happened to have its own `;` before the scope closed -- e.g. `prm = a : 0` immediately followed on the next line by `z = 1/3;` (a real case that surfaced in `test_examples/aac2.inp`) let the second equation's `;` wrongly "terminate" the first, missing-semicolon one. The fix: while scanning forward for a terminator, hitting a *fresh* top-level `=` (a new equation start, same exclusion rules as the outer scan: not part of `==`/`!=`/`<=`/`>=`) before finding `;` now itself counts as proof the current equation is unterminated, and the scan restarts from that new `=` as its own equation. Re-running the full checker across the real 280+ example corpus after this fix surfaced exactly one new, previously-masked FAIL -- `test_examples/cross-corr/cross.inp` line 30, `cross_corr_s = 3 / (1 + .5 Cycle_Iter)` with no trailing `;` before the next line's `chi2 = -Ln(corr); : 0`. This is flagged but deliberately **not** "fixed" by editing that file: it's a TOPAS-shipped example, not a project file, and the manual's own prose example of this exact construct (`references/20-miscellaneous.md`, "Cross correlation function") uses the simpler non-equation form `cross_corr_s 3` rather than this equation form, so whether `cross_corr_s`'s equation form has its own special TOPAS-grammar terminator (ending at the next recognized keyword rather than requiring `;`, the way some reserved-keyword equations might) couldn't be confirmed one way or the other without actually running TOPAS on it. Treat it as a flagged-but-unverified finding, not a confirmed bug -- the same standard this checker asks of every warning it raises.

- `scripts/expand_inp_macros.py` -- expands `macro` calls in an `.inp` file to approximate what TOPAS itself writes to `tc.log` (the literal, fully-expanded text actually fed to the kernel), without needing to run TOPAS. Useful for seeing exactly what a macro-heavy file actually does, or for tracking down a `tc.log` "at LINE N" error against the original macro-based source.
  ```
  python3 scripts/expand_inp_macros.py path/to/file.inp
  python3 scripts/expand_inp_macros.py path/to/file.inp -o expanded.inp
  python3 scripts/expand_inp_macros.py path/to/file.inp --run-number 1   # for files with Run_Number-dependent #if
  ```
  It resolves `#include`s of the `.inc` library (via `TOPAS_DIR`, see "Locating your TOPAS installation"), statically prunes `#define`/`#ifdef`/`#ifndef`/`#if`/`#elseif`/`#endif` branches where it safely can (both branches are kept with a marker comment when it can't), and recursively expands `macro` calls with overload-by-argument-count resolution, parameter substitution (plain text, except for TOPAS's documented `&` syntax -- see below), `@`-parameter auto-naming, and interpretation of the `#m_argu`/`#m_unique`/`#m_first_word`/`#m_ifarg`-`#m_else`-`#m_endif` directive family that TOPAS's own macro bodies use internally to inspect an argument's shape (documented in the script's own docstring, reverse-engineered from real `topas.inc` usage since the manual only mentions `#m_argu` in passing).

  **Implements TOPAS's `&` macro syntax** (manual: "Superfluous parentheses and the '&' Type for macros"), which has two distinct forms, both now handled: `&` before a macro's own NAME (`macro & CeV(c, v) { CV(c, v) }`) wraps that macro's *entire expansion result* in one pair of parens at every call site; `&` before an ARGUMENT name in the parameter list (`macro & Gauss(& xo, & fwhm) { ... }`) wraps *that argument's substituted value* in parens everywhere it's referenced inside the body. Real bug, found and fixed directly from the user's report ("You didnt expand the CeV macro"): the macro-harvesting regex only matched `macro Name(...)`, so the literal `&` in TOPAS's real `topas.inc` definition `macro & CeV(c, v) { CV(c, v) }` broke the match entirely and `CeV` was never recognized as a known macro, leaving every `CeV(...)` call site as unexpanded literal text. Fixed by making the `&` optional in the harvesting regex and tracking a `self_paren` flag per macro definition (applied after `process_macro_invocation` returns, wrapping the whole result), plus threading an `amp_params` set through `parse_param_list`/`process_macro_invocation` so `&`-declared arguments get their bound value pre-wrapped in parens before body substitution. - `scripts/find_refined_params.py` -- **run this whenever the user asks for the independent/refinable parameters in a `.inp` file**.   **Fully macro-expands the file first** (via `expand_inp_macros.py`, imported directly) and scans the EXPANDED text, not the raw file -- this matters because a macro call's bare `@` argument (e.g. `TOF_PV(@, 533.98529, ...)`) doesn't sit textually next to the number it actually binds to; only after expansion does it become an unambiguous `prm AUTO_NAME 533.98529 ...;` statement that can be reliably parsed (confirmed directly: that exact call expands to `prm mb6d5dfaa_0 533.98529\`_13.02250 min .0001 max = ...;`, and `ADPs { @ v1 @ v2 ... }` expands to `load u11 u22 u33 u12 u13 u23 { @ v1 @ v2 ... }`). Several reliable, keyword-independent signals are combined on the expanded text: (1) explicit `prm`/`local NAME ...` declarations -- covers every macro-internal parameter plus anything written directly; `local` is genuinely re-scoped per xdd/phase (manual's "The local keyword": the same name in two different `local` statements is two INDEPENDENT parameters) so these are never deduplicated by name, while bare `prm` has no such re-scoping and IS deduplicated; (2) the `@` sigil on a small curated set of keywords that carry it straight in the kernel and never go through macro expansion at all (`a`/`b`/`c`/`al`/`be`/`ga`/`scale`/`bkg` -- `bkg`'s own convention, only its first coefficient needs the literal `@` while every later space-separated coefficient on that line is independently refined too, handled explicitly); (3) `load u11 u22 u33 u12 u13 u23 { v1 v2 v3 v4 v5 v6 }` -- the `ADPs` macro's own expansion, TOPAS's general `load KW1 KW2 ... { v1 v2 ... }` positional-loading construct (manual section 2.18.1) specialized to the ONE safe case: the full canonical 6-name ADP set with exactly 6 matching value slots -- deliberately not a general `load` parser, since other real `load` forms have entirely different internal structure (`load exclude { 20 22 32 35 }` is flat range-pairs; `load sh_Cij_prm { k00 !sh2_c00 1.0000 ... }` is label/value pairs under one keyword) -- any shape besides the exact 6-ADP-name case is left alone rather than guessed at; (4) a curated (not exhaustive) list of other directly-written NAMED keyword values (`beq`/`u11..u23`/`a`/`b`/`c`/`al`/`be`/`ga`/`scale`, e.g. `beq b1 1.34160`), deduplicated by name for the same "same name = same value, no re-scoping" reason as bare `prm` (Technical_Reference.pdf section 2.4 "Parameter constraints": *"Parameters with the same name must have the same value -- an exception is thrown if [they] were defined with differing values."*).

This is a best-effort static scan, not a full kernel-equivalent parser.

  **A third, much larger gap found and fixed: `for xdds { ... }` / `for strs [N to M] { ... }` loop repetition wasn't accounted for at all -- a parameter declared ONCE in a loop body is really evaluated by TOPAS once per loop iteration, not once total.** Raised directly by the user after comparing this script's count against TOPAS's own reported number on a real 9000+-line multi-pattern file (`test_examples/matthew-rowles/2457_surface_paper_2.inp`, 65 xdd patterns via `for xdds { ... }`): TOPAS itself reported **1448** independent parameters (`tc.exe`'s own `Num independent parameters:` console line, `iters 0`), this script found only **864** before the fix. `check_inp_syntax.find_for_loop_multipliers()` locates every `for xdds { ... }` (multiplier = the file's total xdd count), `for strs { ... }` (multiplier = total str count, but only when NOT itself nested inside another such loop -- see below), and `for strs N to M { ... }` (multiplier = `M-N+1`, unambiguous regardless of nesting) span, and `for_loop_multiplier_at_line()` gives the product of every span containing a given line (nested loops multiply together).

  **Which category actually gets multiplied is NOT "everything inside the loop" -- it's specifically anonymous/unnamed declarations, never bare NAMED ones, confirmed empirically against two minimal test files rather than assumed:** a `for xdds { prm test_named_prm 1 min -2 max 2 }` applied to 2 xdds reported **`Num independent parameters not taking part in refinement: 1`** -- not 2 -- while the exact same test with `prm @ 0.5 min -2 max 2` (anonymous, no name) instead reported **2**. This makes sense once stated: TOPAS's kernel-enforced "parameters with the same name must have the same value" rule (`Technical_Reference.pdf` section 2.4) applies regardless of for-loop context, so a bare NAMED declaration stays ONE shared parameter across every iteration; an anonymous `@` has no name to share an identity by, so it genuinely gets a fresh instance each time. Concretely: `direct_at`/`adp_loads` (always anonymous by construction) and `local` entries (already separately established as genuinely re-scoped per xdd/phase, which a for-loop's once-per-iteration evaluation directly extends) are multiplied; `deduped_named`'s bare-`prm` entries and all of `named_direct` (both always carry an explicit name by construction) are never multiplied, no matter how deep inside a for-loop they sit. Re-verified on the real file after applying this distinction: 864 -> 928 (up from the earlier, over-eager first attempt at 1572, which multiplied everything uniformly and overshot past even TOPAS's own 1448). The remaining 928-vs-1448 gap traces to a *separate*, already-documented limitation above (bare unnamed site `x`/`y`/`z` aren't reported at all) -- confirmed directly on this same file: it uses an empty toggle macro, `macro ref_atom_posn { }`, so `x ref_atom_posn 0.28937 ...` expands to a bare, unnamed `x 0.28937 ...`, invisible to this scan for reasons unrelated to for-loops.

  One deliberately conservative choice, stated plainly rather than guessed at: a bare (no-range) `for strs { ... }` nested inside another for-loop span (almost always `for xdds { ... }`) is left unmultiplied -- its real scope (every str in the whole file again, per outer iteration? or just the current xdd's own strs?) isn't confirmed, and guessing "whole file, again" risks a far worse failure mode (a 65x390 explosion instead of a 65x6) than simply not adding a second multiplier there. `param_dependency_trees.py`'s independent-parameter badge reuses this exact same engine and empirical rule (`Node.scoped`), for the same reason -- see its own entry below.

A site's fractional point is mapped through every space-group operator, reduced mod 1 and deduplicated (its orbit/multiplicity), and any point on a cell face/edge/corner is additionally mirrored to the far side so the box looks visually complete (matching common structure-viewer convention, e.g. VESTA) -- fractional-to-Cartesian uses the standard triclinic cell matrix, so any crystal system renders correctly, not just orthogonal ones.

  Element colors/covalent radii are an approximate built-in table (Jmol/CPK-style colors, Cordero-style radii, ~70 common elements, gray fallback for anything else) -- a readable visual aid, not for scientific bond-length analysis. **Bonds are deliberately restricted to different-element pairs** -- a same-element covalent-radius cutoff was tried first and produced 496 mostly-nonsensical Ce-Ce/O-O bonds on a real fluorite CeO2 test file (a metal/ionic sublattice at normal near-neighbor spacing easily falls inside a same-element covalent cutoff); restricting to different-element pairs dropped that to 256 legible cation-anion bonds and is now the default. Known limitations, stated plainly: occupancy/site-mixing isn't visualized (a mixed-occupancy site is drawn as its first species only); only sites whose x/y/z all resolve to a concrete number are plotted (anything referencing an external `prm` or a more complex equation is skipped with a note); a macro whose `{`/`}` and cell values aren't in one of the forms above (e.g. an equation-based cell parameter) won't resolve.

  **A site whose `u11`/`u22`/`u33`/`u12`/`u13`/`u23` all resolve to concrete numbers (the same simple bare/`@`/same-site-`Get()`-tie forms `resolve_site_coordinates` already handles for `x`/`y`/`z`, extended here by `resolve_site_adps`) is drawn as a real oriented 50%-probability thermal ellipsoid instead of a plain sphere** -- added directly at the user's request ("plot str and include adp ellipsoids") after first converting a `beq`-based site to ADPs via `insert_adps.py`. TOPAS's `u11..u23` are the standard IUCr/CIF "U_cif" convention (referenced to the reciprocal cell edges a*/b*/c*); converted to a genuine Cartesian tensor via the standard nomenclature-report transform `U_cart = A.(N.U_cif.N).A^T` (`A` = the same cell matrix positions already use, `N = diag(a*,b*,c*)`) -- sanity-checked against the trivial cubic case, where `U_cart` reduces to exactly `U_cif` with no correction needed (matching the well-known fact that cubic cells need none). **Each symmetry-equivalent image of a site gets its own correctly reoriented tensor**, not a copy of the original site's ellipsoid: `expand_site_orbit` was extended to also return which space-group operator generated each image (a boundary-mirror duplicate, added purely so the unit-cell box looks complete, inherits its un-mirrored counterpart's operator rather than counting as a new one), and `rotate_u_cart` applies that operator's rotation part in Cartesian space (`R_cart = A.R_frac.A^-1`, then `U_cart' = R_cart.U_cart.R_cart^T`) to each image. Verified directly on a real file (`temp.inp`'s rutile `str`, `P4_2/mnm`, deliberately non-symmetric test ADP values so any rotation would be visible): the origin Ti site's 8 corner-mirror duplicates all correctly share the identical, un-rotated tensor, while the genuine second orbit image (the 4_2-screw-related site) shows `u12`/`u23` sign-flipped and `u13` unchanged relative to the origin -- exactly the signature of a real 90-degree rotation about c, not a coincidence or a no-op.

  The JS side needs no 3D mesh at all for this: the orthographic projection of a 3D ellipsoid onto the view plane is exactly the 2D ellipse given by the top-left 2x2 submatrix of the view-space tensor `R.U_cart.R^T` (`R` from `rotationMatrix()`, built by applying the existing `rotate()` function -- unchanged, still just the two-axis view rotation -- to the standard basis vectors), solved in closed form via `projectedEllipse()` (a symmetric-2x2 eigendecomposition, no iterative solver needed) and drawn with the canvas 2D context's native `ellipse()` method (rotated/scaled via `ctx.rotate()`/an anisotropic radial gradient in local space, same shading style as the existing spheres). A site with only `beq` (or nothing at all) is unaffected -- still the plain covalent-radius sphere exactly as before.

  **Nothing about this is space-group-specific** -- `resolve_sg_operators`/`uij_to_u_cart`/`rotate_u_cart` are plain linear algebra over whatever operators TOPAS's own `sgcom6.exe` resolves for the file's actual `space_group`, and the cell-matrix math is the general triclinic form, not a special case for orthogonal cells -- but this was verified directly, not just asserted, at the user's explicit request ("a python script for this that would work on all space groups with adp parameters"): tested against 5 additional real corpus files spanning cubic (`Pa-3`, `P213`), orthorhombic (`P212121`), monoclinic (`P1211`), and triclinic (`p1`/`p-1`) space groups. That sweep caught one real, common gap: **`resolve_site_adps` only ever tried the bare `u11`/`u22`/... keyword names**, but 63 corpus files (e.g. `sp2/alanine/cif2ta.inp`, `sp2/cholestane/cif2ta.inp`) instead write `ADPs { u11C1 0.01 u22C1 0.01 ... }` -- the site's own tag appended directly onto each ADP keyword, a real, deliberate convention for giving every atom's ADPs a globally unique parameter name (TOPAS enforces "same name = same value" globally, not just per-site, so a many-atom refinement needs each site's `u_ij` distinctly named to stay independent) -- already known to `check_inp_syntax.py`'s own false-positive handling for this exact pattern, just not yet handled here. Fixed by retrying with each name plus the site's own tag appended (`u11` + `C1` -> search for `u11C1`) whenever the bare form doesn't fully resolve, including tie resolution (`u22C1 = Get(u11C1);` needs the tag stripped back off the *referenced* name too before it matches the canonical `known` dict, handled via `_resolve_adp_forms`'s `tag` parameter). Re-verified after the fix: `alanine/cif2ta.inp` (P212121) went from 0 to 24 ADP-ellipsoid atoms, `cholestane/cif2ta.inp` (P1211) from 0 to 109. (Two of the five test files failed for pre-existing, unrelated reasons already covered by this script's stated known limitations -- one needs macro expansion this script doesn't do, one has a `space_group` this script's simple regex scan didn't locate -- not touched here, since neither is about ADPs or space-group generality.)

  **A real, pre-existing correctness bug was found and fixed while regression-testing this change, unrelated to left-vs-right alignment itself**: some real corpus files (`k-factor.inp`, several `mag/` files, and a few others) write a site's fields in a non-canonical order -- `site NAME num_posns # occ ... beq ... x ... y ... z ...`, with `x`/`y`/`z` coming AFTER `occ`/`beq` -- yet `fix_group()`'s rewritten line always follows the canonical `x y z occ beq` order regardless of the source order. The "leave the group's last keyword on each line unpadded" logic used `order[-1]` (that row's own ORIGINAL field order) to decide which field was "last," so for these reordered rows it wrongly picked `z` (last in the SOURCE) even though `beq` is what's actually printed last in the CANONICAL rewritten line -- `z` ended up unpadded mid-line instead of `beq`. Confirmed as a real bug (not just a formatting quirk) by running `fix_columns()` twice on the same file and diffing: the first pass's own reordering changes what "the row's original order" even means, so the second pass computed a *different* padding width than the first -- non-idempotent output, a directly observable defect regardless of alignment direction. Fixed by deriving `last_kw` from `used_keywords` (the canonical order actually being written) filtered to that row's own fields, not from the row's original `order`. Re-verified across the full curated `tcinps-2.bat` corpus (162 files): 0 exceptions, 0 non-idempotent files (down from 6) both before and after this specific fix, confirming it was corpus-real, not synthetic.

  python3 scripts/fix_columns.py file.inp --check      # preview to stdout, don't write

  Whitespace-only -- doesn't validate or change TOPAS semantics, so still run `check_inp_syntax.py` afterward as usual. A site line is tokenized as `site NAME` followed by (keyword, value) pairs, where a value is everything up to the next recognized keyword -- this naturally handles single-token values (`x 0.25`), equation-terminated values (`beq = b1;`), and (via special-casing) `occ`'s two-token `TYPE N` form, which gets its own two independently-aligned sub-columns (`occ Ce+4  1`, single space after the keyword, matching the established corpus convention) rather than being treated as one opaque blob. Consecutive `site` lines (blank lines allowed between them, no other line type) form one alignment group; the group's *last* keyword on each line is left unpadded, since trailing whitespace after it serves no purpose. Verified idempotent (byte-identical output re-running on already-aligned text) and against a real corpus bug: `test_examples/simple.inp` had scrambled site-line spacing (inconsistent gaps, a value split across lines) that this script's logic was built to fix, confirmed via `check_inp_syntax.py` passing clean on the result.

  **A second real bug found and fixed: a keyword-looking word INSIDE an equation's own RHS used to corrupt the whole line.** `test_examples/simple.inp`'s O2 site has `z = Get(x) + 0.1625;` -- the original tokenizer did one blind `KEYWORD_SPLIT_RE.split()` over the whole line with no concept of an equation's extent, so the bare `x` inside `Get(x)` was mistaken for a second `x`-keyword boundary; since `fields` is a plain dict keyed by keyword name, that second match silently overwrote the real `x` value, corrupting the whole line end to end (observed directly: running the formatter on this exact file produced `site  O2   x  ) .99 + 0.1625;  y  0.375  z  = Get(  occ O-2   1  beq = b2;` -- the real `x`/`z` values swapped into garbage, with no error raised). `parse_site_line()` now finds keyword matches via `finditer` and explicitly skips any match that falls inside an *earlier* keyword's own equation body (between its `=` and that equation's terminating `;`), so a `Get(x)`/`Get(y)`/`Get(z)` reference inside a `z`/`y`/`x` equation is never mistaken for a new keyword boundary. Omit `--lines` to fix columns through the whole file, matching prior behavior. When the user has active IDE selection context (an `<ide_selection>` block naming a line range) and asks to fix columns, translate that range directly into `--lines START-END` rather than fixing the whole file.

  Only keywords with a non-empty resolved children set become section-openers (this naturally covers `xdd`, `str`, `site`, `hkl_Is`, `xo_Is`, `d_Is`, `lam`, and others, without hand-listing them) -- everything else (`scale`, `bkg`, `prm`, `axial_conv`, `cell_mass`, `space_group`, ...) is ordinary content that never itself changes the indent level. A section-opener closes any open section it isn't a recognized child of, but "recognized child of" resolves two different ways depending on how open-ended the section is -- also derived from the schema, not hand-picked per keyword: a **wide** section (its resolved children reach `prm`, i.e. TOPAS's general "any refinement content is valid here" `Tcomm_2` mixin -- true for `xdd`/`str`/`hkl_Is`/`xo_Is`/`d_Is`) is only closed by *another* section-opener it doesn't recognize as a child, so ordinary content never closes it (a second `str` becomes a sibling of the first; a new `xdd` resets all the way back). A **narrow** section (no `prm` reachable -- true for `site` and `lam`, each with a small fixed field list and nothing else) is closed by *any* keyword at all that isn't one of its own children -- e.g. `site`'s `adps`/`u11` lines don't leak into the next `site`, and `lam`'s `la`/`ymin_on_ymax` lines don't leak into the `axial_conv` that follows a `lam` block.

## Core syntax cheat sheet

- Blocks are delimited with `{ }`; comments use `'` to end-of-line.
- A parameter gets a name (letters, digits, underscore, must not start with a digit) so it can be tracked/refined; prefix `!` to fix it (exclude from refinement); prefix `@` to auto-generate a unique internal name.
- `prm name value` declares a standalone named parameter; combine with attributes like `min`, `max`, `update`, `stop_when` as needed.
- **Prefer the subject-specific keyword over a bare `prm` for ANY refinable value that has one -- this is a general rule, not just for lattice parameters/`beq`/`occ`.** Confirmed directly in `topas.inc` itself: essentially every macro that declares a `prm` for a genuinely refinable physical quantity (via the shared `If_Prm_Eqn_Rpt` helper used throughout, e.g. in `CS_L`, `CS_G`, `LP_Factor`, `Zero_Error`, `TOF_PV`, and dozens more) supplies explicit `min`/`max` (or at least a `min` floor, for quantities like FWHM with no natural upper bound) right alongside it -- the library's own author never leaves a refinable `prm` bare, and neither should you. The exceptions that show up in `topas.inc` (a bare `prm` with no min/max) are internal bookkeeping/scratch values that are never actually refined, not physical parameters. Many subject-specific keywords additionally carry a documented default `min`/`max` (full list in `references/01-syntax-and-parameters.md`'s Table 2-1 -- e.g. lattice parameters default to roughly ±0.5% of their starting value, `beq` defaults to `Max(-10, Val-10)`/`Min(20, Val+10)`), which going through a bare `prm` + equation indirection silently throws away regardless of which physical quantity it is. **This applies just as much when manually expanding a macro to raw keywords** (e.g. answering a "write this without macros" request) -- the macro was silently supplying `min`/`max` alongside its `prm`, and expanding it to a bare `prm` drops that bound even though nothing about the request asked for it to be dropped. If `topas.inc` isn't available to read the macro's actual bound (no `TOPAS_DIR` set, see "Locating your TOPAS installation"), don't leave the parameter unbounded and don't silently invent a number that looks authoritative either -- add a physically-reasoned placeholder bound and say plainly that it's a placeholder pending the real value, the same way you'd flag any other unverifiable specific. For a cubic cell with `a` refined:
  ```
  a prm_name 4.56
  b = Get(a);
  c = Get(a);
  ```
  not
  ```
  prm prm_name 4.56
  a = prm_name;
  b = prm_name;
  c = prm_name;
  ```
  If a bare `prm`/`local` genuinely is the right tool (no subject-specific keyword fits, e.g. a scratch value used only in an equation), define its own `min`/`max` explicitly rather than relying on TOPAS's generic fallback.
- Equations start with `=` and end with `;`, e.g. `beq = 0.5 + b1;`. Anything to the right of `=` is evaluated as an equation and can reference other named parameters. The `;` is mandatory even when the equation is immediately followed by a `:` value-report (see next bullet) -- `prm bb = cs1 + cs2 : 0` (no `;`) is invalid; it must be `prm bb = cs1 + cs2; : 0`. See `references/01-syntax-and-parameters.md` § "Reporting on equation values"/"Naming of equations".
- A trailing `: 0` (or any placeholder number) after an equation's terminating `;` tells TOPAS to report that equation's evaluated value back into the file in place of the placeholder once refinement finishes -- e.g. `occ Ti+4 = 1-zr; : 0` gets the `0` overwritten with the refined value (plus its error, if `do_errors` is set) after the run, the same way a refined `prm`'s value gets written back with a trailing `` `_err ``. This also works on a bare, unnamed equation (`prm = 2 a1^2 + 3; : 0;` -- valid without a parameter name, purely to report a derived value) and is exactly the pattern already sitting throughout `test_examples/aac2.inp`'s `prm bb = cs1 + cs2; : 1088.36411\`_7.84708627` line -- the reported value there is TOPAS's own output from a previous run, not something to hand-edit.
- Structures (`str { ... }`) contain `site` definitions with `x y z occ beq` etc.; intensity-only phases use `hkl_Is` or `xdd_Is` instead of a full structure.
- **`al`/`be`/`ga` default to 90 degrees if omitted from a `str` entirely** -- confirmed directly by the user (TOPAS-Academic's author). This holds regardless of `space_group`, including `P1`, so don't write `al 90 be 90 ga 90` just because the space group label no longer implies it -- if the cell genuinely is orthogonal, leaving the angle keywords out gets the same result. See `references/01-syntax-and-parameters.md` § "Default lattice angle value" for the full note.
- Macros: `macro Name(args) { ... }`, invoked as `Name(actual_args)`. Prefer using or adapting an existing macro pattern from the references over writing raw repeated blocks.
- **Converting a binary data file (`.raw`, etc.) to plain-text `.xy`: use TOPAS itself, via `Out_X_Yobs`, rather than writing a custom parser or guessing that a same-named `.xy` file already sitting nearby is equivalent.** A minimal conversion INP, confirmed working directly by the user (TOPAS-Academic's author) and verified byte-for-byte against a real file (`test_examples/pbso4.raw` → `pbso4.xy`, 6001/6001 matching lines):
  ```
  iters 0
  xdd pbso4.raw
  Out_X_Yobs(pbso4.xy)
  ```
  `iters 0` skips refinement entirely (no structure/parameters needed — this is a pure format conversion, not a fit); `Out_X_Yobs(file)` (documented in `references/06-macros-and-include-files.md`) writes the x-axis and Yobs columns to `file`. This works for any format TOPAS's `xdd` can read natively (Bruker RAW, GSAS, etc.) — no need to hand-write a binary-format decoder, and no need to rely on file-size arithmetic or a same-named file's presence as circumstantial evidence that it matches. Run via `tc.exe path/to/conversion.inp`.

## Common errors to check for when debugging

- Missing semicolon at the end of an equation (`= ...;`) — TOPAS equations always terminate with `;`. This is especially easy to drop right before a `:` value-report (`= cs1 + cs2 : 0` instead of `= cs1 + cs2; : 0`) — the `:` reads visually like it could be its own clause, which makes the missing `;` easy to miss on a skim. A real instance of exactly this was found and fixed in `test_examples/aac2.inp` during development of this skill.
- Unbalanced `{ }` — TOPAS is whitespace-insensitive but brace-matching errors are the most common source of cryptic parse failures.
- Forgetting to flag a parameter for refinement (no name, or accidentally leaving `!` on a parameter that should vary).
- Declaring ANY refinable value with a bare `prm` + assignment equation instead of its subject-specific keyword (not just lattice parameters/`beq`/`occ` — this applies universally) — this quietly loses that keyword's built-in default `min`/`max` attributes where one exists, and more generally departs from the convention `topas.inc` itself follows everywhere (see the cheat sheet above and Table 2-1 in `references/01-syntax-and-parameters.md`). If you spot this pattern while debugging, check whether the refinement is wandering unbounded because of it before assuming the problem is elsewhere.
- Using `str` when the phase should be `hkl_Is`/`xdd_Is` (or vice versa) — this changes which keywords are valid and how intensity is calculated.
- Referencing a parameter name before it's defined, or reusing a name across scopes unintentionally (macros help avoid this).
- Mismatched data structure context for a keyword — verify against `references/21-keyword-index.md` which structure (`Ttop`, `Tglobal`, `Txdd`, etc.) a keyword actually belongs to before assuming it's valid where it's being used.
- A dependent file from an earlier stage doesn't exist yet (e.g. an `.fc` file a later run expects) — TOPAS reports this as `Cannot open file ...` followed by `Abnormal program termination.`; check whether a prerequisite run needs to happen first rather than assuming the current INP itself is broken.
- An invalid reserved-parameter/keyword combination — TOPAS reports the exact line number and the specific name it couldn't resolve (e.g. `Cannot locate X from Y in data structures`). See `references/console-output-and-errors.md` for real examples of both of these and what actually caused them.

## Stripping refined-value errors (` `_err `) from an INP

A common cleanup request ("remove errors" — a standing trigger phrase) is removing every trailing `` `_<error> `` suffix from refined values (e.g. `48.0611197`_1.21585437` → `48.0611197`) so a file with fitted results reads like a fresh, unrefined starting INP. The instances scattered through a file never share a common literal substring (each has a different numeric error), so a text-editing tool limited to exact literal `old_string`/`new_string` matching (no regex/wildcards) can't do this in one call — it forces one call per occurrence, which is slow and easy to under- or over-match on a long file.

**Use `scripts/remove_errors.py`** (a single regex pass, `` `_-?\d+\.?\d*(?:[eE][-+]?\d+)? `` — the same pattern this section used to document as a manual PowerShell one-liner before this script existed) rather than hand-running a shell regex each time:

```
python3 scripts/remove_errors.py file.inp                 # rewrite in place, whole file
python3 scripts/remove_errors.py file.inp -o out.inp       # write elsewhere
python3 scripts/remove_errors.py file.inp --check          # preview to stdout, don't write
```

Prints how many suffixes were removed either way. Verify afterward with `scripts/check_inp_syntax.py` that the result still parses cleanly.

## Peak-picking and peak-fitting for indexing (visual method)

When no automated Peak-Search result is available and peaks need to be picked by eye from a plotted pattern before a peak-fit → indexing pipeline (see `pbso4.raw`/`pbso4.xy` in `test_examples/` for the full worked example this section is drawn from), this is an iterative process, not a one-pass eyeball-and-done — confirmed directly and corrected twice by the user (TOPAS-Academic's author) after an initial pass missed real peaks.

**1. Pick peaks at the best resolution available, not from a wide overview.** A single plot spanning 30-35 degrees at once visually compresses real, distinct peaks into what looks like a single bump or an ignorable shoulder — a first attempt at this exact pattern, viewed at that scale, missed 6 real peaks (38.13, 40.24, 42.32, 47.67, 53.30, 57.08 degrees) and misread one region as a single peak when it was genuinely two (44.5/44.6 degrees). Re-examining the same data in ~5-degree-wide windows with fine (0.1 degree) gridlines caught all of them. Plot the *entire* candidate range this way (not just a region that already looks interesting) before trusting a peak list as complete.

**2. Distinguish a genuine second peak from a Cu Kα1/Kα2 instrumental doublet before adding it.** Lab XRD data using a Kα-doublet wavelength macro (`CuKa2_analyt`, `CuKa5`, etc.) already models the Kα1/Kα2 split *within a single peak entry* — confirmed by reading the macro's own definition in `topas.inc` (`CuKa2_analyt` resolves to `Load_Lam(cuka2_analyt, ...)`, a two-line wavelength table weighted ~0.66/0.34). A shoulder immediately next to a strong peak is very often this splitting, not a second reflection — check by predicting the Kα2 companion's position via Bragg's law: `sin(θ_Kα2) = sin(θ_Kα1) × (λ_Kα2 / λ_Kα1)` (λ_Kα2/λ_Kα1 ≈ 1.002485 for Cu), which always places the Kα2 companion at *higher* 2θ than its Kα1 partner, by an amount that grows with angle (a few hundredths of a degree around 20 degrees 2θ, growing to roughly 0.1-0.15 degrees by 45 degrees, confirmed against several real shoulders in `pbso4.xy`). A shoulder matching that predicted direction and magnitude should be left for the wavelength macro to handle, not added as a separate peak (double-counting the same reflection). A shoulder on the *wrong side* (lower 2θ than the main peak) cannot be Kα2 splitting and is real evidence of a second reflection — this is exactly how the genuine 44.5/44.6 degree doublet in `pbso4.xy` was confirmed as two peaks rather than one.

**3. After fitting the picked peaks, generate the calculated pattern with `Out_X_Ycalc(file)` (documented in `references/06-macros-and-include-files.md`) and overlay it against the observed data — do not trust the fit's Rwp alone.** Add the line inside the `xdd { }` block after the `xo_Is`/`load xo I {}` block:
```
Out_X_Ycalc(output.xy)
```
Then plot both `output.xy` (calculated) and the original observed `.xy` on the same axes, in the same fine (~5 degree) windows used for picking. **The overlay should show no observed peak without a matching calculated one** — any blue (observed) bump the red (calculated) curve doesn't track is a peak that was missed during picking (or, less often, a peak-shape/width problem worth a closer look). Add any newly-found peaks to the `load xo I {}` list and refit; repeat until the overlay shows no unmatched observed peaks anywhere in the fitted range. This step is what caught the 6 missed peaks above — a first peak-fit had a deceptively reasonable Rwp (30.1%) despite missing real peaks, because Rwp alone doesn't localize *where* a fit is wrong the way a direct visual overlay does.

**4. Restrict `start_X`/`finish_X` to the region that actually has usable peaks, and stop `finish_X` *before* peaks become poorly resolved, not just wherever intensity happens to drop off.** `start_X`/`finish_X` go inside the `xdd { }` block, e.g.:
```
xdd pbso4.raw
   start_X 15.94
   finish_X 65
   ...
```
Pick `start_X` a few tenths of a degree before the first real peak and `finish_X` a few tenths after the last one intended for fitting — this alone improves Rwp substantially by removing large featureless regions from the weighted residual (26.4% → 11.6% Rwp on the same peak list, purely from tightening the range). For the upper cutoff specifically, look at the fine-resolution plot of the region just past the intended `finish_X`: if peaks there are still individually distinguishable (a clear rise-peak-fall shape with real separation between neighbors), the range can extend further; once peaks visibly blend into each other (multiple overlapping small bumps with no clean baseline between them), that's the boundary — stop `finish_X` there, since peaks past that point can't be reliably picked or fit as individual reflections anyway. In the worked `pbso4.xy` example, peaks below 65 degrees are all cleanly separated, while 65-70 degrees is a dense, poorly-resolved cluster — confirming 65 degrees as the right cutoff, not an arbitrary round number.

**The full loop, in order**: plot at high resolution across the whole candidate range → pick peaks → build/refit a peak-fit `.inp` with `start_X`/`finish_X` bracketing just the picked peaks → add `Out_X_Ycalc` and re-run → overlay calculated against observed at the same high resolution → fix any unmatched observed peaks (add + refit) or confirm the range boundary is placed before peaks overlap → repeat steps until the overlay is clean → *then* feed the final refined peak list into indexing (`load index_th2 index_I {}`, see the existing indexing example files).

## Building a supercell (enlarging the unit cell by an integer factor)

Two non-obvious things came up building a real example (`test_examples/simple.inp`, CeO2 doubled ×2 in each direction while keeping `space_group F_M_3_M` rather than dropping to `P1`) — both verified by actually running `tc.exe`, not derived from the manual alone, so treat the reasoning as confirmed-by-testing rather than documented TOPAS behavior:

- **`scale` must be divided by the volume ratio squared, not left unchanged.** Doubling `a`/`b`/`c` (volume ×8 for a ×2×2×2 supercell) does **not** preserve the calculated pattern if `scale` is left at its original-cell value — an `iters 0` sanity check came back with Rwp ≈ 5827% (badly wrong) using the unchanged `scale`. Letting `scale` refine freely (everything else held at the small-cell's converged values) converged to `original_scale / 64` (64 = 8², i.e. `scale_new = scale_old / (V_new/V_old)^2`) almost exactly, and using that value directly reproduced the original Rwp (9.28% vs 9.23%) at `iters 0`. A first-principles attempt to predict this from `|F|^2/V^2` scaling arguments predicted *no* change was needed and was wrong -- don't trust that derivation over this empirical result. `MVW`'s `cell_mass`/`cell_volume` arguments need the same volume-ratio scaling (×8 for a ×2×2×2 supercell) for the reported quantities to stay correct, though this doesn't affect the fit itself.
- **Keeping the original (higher-symmetry) space group instead of expanding to `P1` requires finding the space group's actual symmetry orbits, not just listing every supercell atom.** Space-group symmetry operators will auto-generate duplicates if you list all N supercell-expanded atoms directly under the original space group. The correct approach: compute the space group's real symmetry operators (point-group operations × centering translations -- cross-check the operator count against what TOPAS itself reports, e.g. `Num equiv posns for centrosymmetric F_M_3_M: 192`), partition the full supercell atom-position set into orbits under those operators, and use one representative per orbit as the explicit `site`. **Verify the result with `Out_CIF_STR`** before trusting it -- it dumps TOPAS's own expanded atom list with each site's actual multiplicity, which should exactly match the orbit sizes computed independently (confirmed exactly for the CeO2 case: 24+4+4 Ce, 32+32 O). This is a much more compact description (5 explicit sites instead of 96 here) and is the same reasoning already documented for the `P1`-vs-higher-symmetry angle-default question in `references/01-syntax-and-parameters.md` § "Default lattice angle value" -- lower the symmetry label only when you actually need to (e.g. genuinely broken symmetry), not as a default way to sidestep working out the space group's orbits.

If debugging a supercell that doesn't reproduce the original pattern, isolate which half is wrong before assuming the atom positions are broken: build a parallel `P1` version with every atom listed explicitly and compare its `iters 0` Rwp against the reduced-space-group version. If they match (both wrong, or both right), the bug is in something shared (like `scale`), not the site-generator/orbit reduction.

## Converting a CIF file to `str` format

**Preferred: `python3 scripts/cif_to_str.py input.cif`** -- a Wyckoff-constraint-aware converter built for this skill, not a hand-derived field-by-field mapping and not `cif1.exe` (see below for why). It derives each site's coordinate constraints directly from the CIF's own `_symmetry_equiv_pos_as_xyz` operator loop -- no external space-group table needed -- and correctly distinguishes three cases per coordinate:

- **Free** (general position, or the one independent coordinate of a partial special position): emitted as `x @ 0.34131` (refined).
- **Fixed** (a symmetry operator pins it to an exact constant): emitted as a bare value, or as an exact `z = 5/6;` equation when it snaps to a common fraction (see below for why exact form matters).
- **Tied to another coordinate** of the *same* site (e.g. a site sitting on a mirror or a 3-fold axis): emitted as `y = Get(x);` -- confirmed directly by the user (TOPAS-Academic's author) that `Get(keyword)` inside a site's own equation resolves to that same site's other keyword value with no separate naming needed, the identical mechanism the `Cubic(cv)` macro already uses internally (`b = Get(a); c = Get(a);`).

This matters because a hand- or tool-converted CIF that just writes three independent `x @ / y @ / z @` values for every site is silently wrong whenever a site sits on a special position: refinement is then free to pull tied coordinates apart, breaking the very symmetry the CIF's own data already obeys. Validated against three real corpus cases, each cross-checked by actually running the output through `tc.exe` and `Out_CIF_STR` (not just eyeballing the Python):

- `test_examples/ceo2.cif` (Ce/O both fully fixed, F_M_3_M) -- reproduces the already-verified fully-fixed result exactly.
- `test_examples/pdf-adps/je1/zrw2o8_500k_riet_uij.cif` (Pa-3) -- in one file: `Zr1`/`O3` fully fixed, `O1` fully general (all three free), and `W1`/`W2`/`O4` on the body-diagonal 3-fold axis where site symmetry forces `x=y=z`. The converter correctly avoids emitting a circular `x=Get(y); y=Get(z); z=Get(x);` (an earlier version of this script did exactly that and was caught only by actually running it through `tc.exe`) -- instead it picks one coordinate as the free representative and ties the other two to it (`x @ 0.3413131  y = Get(x);  z = Get(x);`). `Out_CIF_STR` round-tripped this back to `x=y=z` exactly, with multiplicities (4/8/24/4/8) matching the source CIF precisely.
- `test_examples/Kate/GaPO4/NeutronPDF/GaPO4_neutron_NOMAD.cif` (P3121) -- `Ga`/`P` each have `y` fixed to 0 and `z` fixed to a different constant (1/3 or 5/6) via an operator whose equation constrains a coordinate *other than* the one its row nominally belongs to (`x-y` in the x-output row is actually a constraint on `y` alone) -- this is the case that required deriving the general `(R-I)p ≡ -T (mod 1)` equation per operator row rather than a naive "look at row i, assume it constrains coordinate i" heuristic.

**Precision matters here, not just correctness of which case applies.** The first version of this converter wrote fraction-snapped fixed coordinates as 6-significant-digit decimals (`z 0.833333`); running that through `tc.exe` produced a real warning from TOPAS itself (`Site P ... has one of its equivalent positions at a distance of 7.37e-06 Angstroms ... Recurring fractional atomic coordinates such as 0.3333 should be entered as x = 1/3;`) -- i.e. TOPAS's own equivalent-position check flagged the rounding error. Fixed by emitting the exact `N/D` fraction as an equation (`z = 1/3;`) whenever a coordinate snaps to one, rather than a rounded decimal -- confirmed the warning disappears and the `Rwp` is bit-identical. This is also exactly the pattern `check_xyz_near_one_third` (below) exists to catch on hand-written or externally-converted `.inp` files -- run that checker on any CIF-derived `str` block regardless of which converter produced it.

**Lattice angles crystallographically forced to exactly 90° are omitted entirely, relying on TOPAS defaulting an absent `al`/`be`/`ga` to 90°** (raised directly by the user, TOPAS-Academic's author). Determined the same way as the coordinate constraints -- from the rotation parts of the CIF's own symmetry operators, no external space-group/crystal-system table -- via `determine_fixed_angles()`: counting proper 3-folds (>=8 means cubic, all three angles fixed), checking for a 4-fold (tetragonal) or a single 3-fold axis (trigonal/hexagonal, which fixes `al`/`be` but *not* `ga`, since hexagonal's 120° isn't TOPAS's default and must stay explicit), and otherwise counting axis-aligned 2-folds/mirrors (three distinct axes = orthorhombic, all three fixed; one axis = monoclinic, the two angles touching that axis fixed, the oblique one left explicit). A fixed angle is only actually omitted if the CIF's own stated value for it is also within tolerance of 90° -- guards against ever silently dropping an angle on a misclassified or non-standard-axes case. Verified via `tc.exe`: omitting `al 90 / be 90 / ga 90` on the cubic `ZrW2O8` case, and omitting just `al`/`be` while keeping `ga 120` explicit on the trigonal `GaPO4` case, both reproduced bit-identical `Rwp` to the fully-explicit version with zero errors/warnings.

**Lattice *lengths* forced equal by the crystal system get the identical `Get()`-tie treatment as tied site coordinates** (raised directly by the user, TOPAS-Academic's author, with the exact worked example: `"a SOME_VALUE  b = Get(a);  c = Get(a);"` -- literally the same pattern the built-in `Cubic(cv)` macro already uses internally, `a cv  b = Get(a);  c = Get(a);`). `classify_crystal_system()` (shared with the angle logic above) drives `determine_length_ties()`: cubic ties both `b` and `c` to `a`; tetragonal/hexagonal-axes-trigonal tie only `b` to `a` (`c` stays independent); orthorhombic/monoclinic/triclinic tie nothing (no length relationship in the standard axis setting). As with the angle check, a tie is only actually applied if the CIF's own stated lengths already agree to within a relative tolerance -- never silently forced. Verified via `tc.exe` + `Out_CIF_STR`: the cubic `ZrW2O8` case (`a 9.136064  b = Get(a);  c = Get(a);`) resolved to `a=b=c=9.136064` exactly, and the trigonal `GaPO4` case (`a 4.9053  b = Get(a);  c 11.0557`) resolved to `a=b=4.9053, c=11.0557` exactly -- both bit-identical `Rwp` to the fully-independent version, zero errors.

**Confidence across all space groups, and a real gap found by checking `sgcom5.txt` (TOPAS's own space-group generator database, asked about directly by the user):** the method generalizes by *crystal class*, derived from the CIF's own operators, not from a hardcoded per-space-group table -- so it isn't limited to the handful of space groups actually tested. But it has only been empirically verified end-to-end (via `tc.exe`) against 3 real CIFs (`ceo2.cif`, cubic; `zrw2o8_500k_riet_uij.cif`, cubic; `GaPO4_neutron_NOMAD.cif`, trigonal in the standard hexagonal-axes setting) -- not against all 230 space groups or every alternate setting `sgcom5.txt` lists (e.g. the orthorhombic/monoclinic axis permutations `abc`/`ba-c`/`cab`/`-cba`/`bca`/`a-cb` its own header comment documents). Reading `sgcom5.txt` directly surfaced one confirmed real gap: it carries a **separate rhombohedral-axes entry for every rhombohedral space group** (`r3`/`r3r`, `r-3`/`r-3r`, `r32`/`r32r`, `r3m`/`r3mr`, `r3c`/`r3cr`, ...) -- a setting where `a=b=c` and `al=be=ga` (mutually equal, but generally *not* 90 degrees), which a single 3-fold axis's rotation matrices alone can't distinguish from the standard hexagonal-axes setting (`a=b`, `c` independent, `al=be=90, ga=120`). Fixed by disambiguating via the CIF's own angle values in `determine_length_ties()`: if al/be/ga are mutually equal but don't match the hexagonal (90, 90, 120) pattern, it's the rhombohedral-axes case, and `c` gets tied to `a` as well (`b=Get(a); c=Get(a);`), not just `b`. Deliberately scoped to lengths only -- the three oblique angles being tied to *each other* (not just individually free) is a real, symmetry-true fact about rhombohedral axes that this script does not yet express, left as a known gap rather than guessed at. Verified with a synthetic operator set (the same 3-fold cyclic-permutation matrices already confirmed correct for `ZrW2O8`, since a rhombohedral 3-fold axis is the same body-diagonal rotation) since no real rhombohedral-axes CIF exists in `test_examples/` to test against -- the hexagonal-axes sanity check (still ties only `b`) was re-run alongside it to confirm no regression. This class of gap -- a rotation-only classification being ambiguous between two real settings that only the actual cell values can disambiguate -- is the reason to treat "confident for the crystal systems tested" and "confident for all 230 space groups" as different claims; don't overstate the latter without either testing further real CIFs across more settings or reasoning through `sgcom5.txt`'s alternate-setting list the way this gap was found.

**Fallback when the CIF itself has no `_symmetry_equiv_pos_as_xyz` loop: resolve operators via TOPAS's own space-group database (`sgcom6.exe` / the `sg/` directory), not a guess** -- raised directly by the user (TOPAS-Academic's author), including the correct standalone `sgcom6.exe` invocation syntax after an initial guess at it was wrong: `sgcom6 SYMBOL -dir sg` (must be run with cwd = `TOPAS_DIR` -- it looks for `sgcom5.txt` relative to its own working directory, confirmed directly: any other cwd fails with `Cannot open file sgcom5.txt`). `resolve_sg_operators()` in `cif_to_str.py` first checks whether `TOPAS_DIR/sg/<symbol>.sg` already exists (the common case -- over 100 are typically already generated from prior real refinement runs in this environment) and reads it directly; otherwise it generates it with the command above. A `.sg` file's `xyzs { }` block is in the *exact same* textual operator format as a CIF's own loop (confirmed identical, line-for-line, comparing `zrw2o8_500k_riet_uij.cif`'s own 24-operator Pa-3 list against `sg/pa-3.sg`'s), so the same `parse_symop_string()`/constraint-derivation pipeline handles both sources unchanged -- only comment lines inside `xyzs` (`' +(-1/3, 1/3, 1/3) ---`, documenting centering-translation groups) need skipping, handled by `parse_sg_file()`. One filename quirk found empirically: `sgcom6.exe` can't put a literal `/` in a Windows filename, so it substitutes `o` (`p21/n` -> `p21on.sg`) -- the `symbol` field stored *inside* the file keeps the real `/`. This fallback is best-effort, not guaranteed: the symbol must already be in the concise form `sgcom6.exe` recognizes (`fm-3m`, `p21/n`), not CIF's underscore-spaced style (`F_M_3_M`) -- a failed resolution degrades to the same "no constraints derivable, verify manually" per-site warning as before, never a crash. Confirmed working end-to-end on a synthetic test (`GaPO4_neutron_NOMAD.cif` with its own operator loop stripped out, symbol `P3121` only): the fallback-resolved output was **identical** to parsing the CIF's own operators directly. Every resolved-via-fallback conversion carries an explicit warning naming the `.sg` file used, since the constraints then come from TOPAS's own database keyed by the symbol string, not from the CIF's own self-contained data -- worth a closer look if the CIF's symbol could plausibly resolve to an unintended setting.

**Anisotropic displacement parameters (`u11`/`u22`/`u33`/`u12`/`u13`/`u23`) get the same site-symmetry-constraint treatment as coordinates -- raised directly by the user (TOPAS-Academic's author): "depending on the type of Wyckoff site, can you generate atomic displacement parameters and replace the adps keyword with... u11, u22, u33, u12, u13 and u23. Put constraints on the unn parameters as required."** TOPAS's own bundled `adps` keyword already does exactly this at runtime -- `references/21-keyword-index.md`: *"adps generates the unn atomic displacement parameters with considerations made for special positions... On termination of refinement the adps keyword is replaced with the unn parameters"*, using "the 3x3 eigenvalue determination routine of Kopp (2006)". `classify_adps()` in `symmetry_utils.py` is an independent, ahead-of-refinement derivation of the same result -- useful for converting a CIF's own `_atom_site_aniso_U_11..23` values directly, without needing an actual `adps`-then-refine-then-inspect round trip through TOPAS.

The method is the tensor analogue of `classify_coordinates`: a site's ADP tensor U must satisfy `R U R^T = U` for every rotation R in its stabilizer (translation is irrelevant -- a tensor property at a fixed point doesn't depend on it, unlike a coordinate). `adp_transform_matrix()` builds the 6x6 matrix mapping the 6 unique tensor components `[u11,u22,u33,u12,u13,u23]` under this transformation (verified by transforming each of the 6 symmetric basis matrices directly, not a hand-derived index formula); `classify_adps()` accumulates `(T-I)u=0` across every non-identity stabilizing operator and solves the combined homogeneous system via Gaussian elimination over exact `Fraction` arithmetic (`_rref()`) -- unlike the position case, a hand-rolled pairwise tie model isn't sufficient here, since a single 3-fold rotation already mixes 3 components (e.g. `u11`/`u22`/`u12`) in one equation. Pivot columns are sought in reverse name order (`u23,u13,u12,u33,u22,u11`) so diagonal `Uii` terms stay free/independent where there's a choice, matching crystallographic convention. A component with no free-variable dependence is always fixed at **exactly zero** (unlike position's `'fixed'` case, ADP constraints are purely homogeneous -- there's no tensor equivalent of a coordinate's special-position constant).

Verified end-to-end against real experimental data, not just internal consistency: `ZrW2O8`'s own CIF (`pdf-adps/je1/zrw2o8_500k_riet_uij.cif`) carries a real `_atom_site_aniso_*` loop, and every site on the Pa-3 body-diagonal 3-fold axis (`Zr1`, `W1`, `W2`, `O3`, `O4`) shows `U11=U22=U33` and `U12=U13=U23` in the CIF's own refined numbers -- `classify_adps()` derives exactly `u22 = Get(u11); u33 = Get(u11); u13 = Get(u12); u23 = Get(u12);` for each of them (`O1`, the one general-position site, correctly comes out fully free, all six independent). Running the generated `str` block through `tc.exe` with `Out_CIF_ADPs` and reading back the resolved tensor confirmed **exact numeric agreement with the original CIF** for all six sites -- not approximate, bit-for-bit the same six decimal values TOPAS itself originally refined to.

`format_adp_tie()` handles the general case of a tie's coefficient not being ±1 (e.g. a hexagonal-axes-along-c site symmetry can produce `u12 = Get(u11) / 2`) and multi-term linear combinations (rare, but the general linear-algebra derivation can produce one, e.g. `u23 = Get(u12) + Get(u13)`), which the position-coordinate model's simpler pairwise ties never needed to express.

**A real bug caught by testing against a second real CIF, not just ZrW2O8: an all-zero `_atom_site_aniso_*` row is a common unused/placeholder pattern, not real data, and must not be trusted at face value.** `GaPO4_neutron_NOMAD.cif` carries an aniso loop, but every value in it is a literal `0` (`Ga 0 0 0 0 0 0`) -- meaningless boilerplate from whatever software produced the file, while the *real* thermal-motion data for this structure lives in `_atom_site_B_iso_or_equiv` (genuine nonzero values: 0.912, 1.08, 1.39, 1.31). An early version blindly trusted the aniso loop's presence and emitted a physically meaningless all-zero ADP tensor (correctly Get()-tied, but tied-together zeros are still zero), silently discarding the real isotropic data. Fixed: a site's aniso row is only used if not all six components are exactly zero (within `1e-12`); an all-zero row falls back to `beq` instead, with a warning explaining why. Re-verified both cases after the fix: `ZrW2O8` (real nonzero aniso) is unaffected, bit-identical to the already-`tc.exe`-verified output; `GaPO4` now correctly emits `beq` instead of a false all-zero tensor.

Known limitations of `cif_to_str.py` (be upfront about these when using its output):
- Coordinate-tie detection assumes the standard crystallographic case (integer -1/0/1 rotation-matrix entries); a constraint row with 3+ nonzero entries, or a 2-entry row where neither coefficient is ±1, is flagged `'complex'` (emitted as an independent value with a `' verify manually` comment) rather than guessed at.
- If `_atom_site_type_symbol` is missing, it guesses the element from the site label (stripping trailing digits, e.g. `O1`→`O`) and emits a warning -- verify this against the actual CIF chemistry, it is not always correct (see the GaPO4 case above).
- Cross-checks the CIF's stated `_atom_site_site_symmetry_multiplicity` against the independently-computed orbit size; a mismatch scales the occupancy and emits a warning (possible disorder/split site, or coordinates not landing exactly on the intended special position) rather than silently trusting either number.
- Not a general CIF-dictionary parser -- built for the specific constructs a `str` conversion needs (cell, symmetry operators, atom sites). Unusual CIF formatting may not parse; always verify the output (e.g. via `check_inp_syntax.py` and a real `tc.exe` run with `Out_CIF_STR`, as above).
- The `sgcom6.exe` fallback needs `TOPAS_DIR` set and a real install present -- without it, a CIF missing its own operator loop falls back further to the old "can't derive constraints" per-site warning, same as before this fallback existed.
- ADP handling assumes an all-zero `_atom_site_aniso_*` row means "unused placeholder, use `beq` instead" (see the `GaPO4` case above) -- correct for every real case seen so far, but a genuine (if physically unusual) refinement that converged to isotropic-zero anisotropic displacement would be misread the same way; a nonzero `beq`/`B_iso_or_equiv` alongside an all-zero aniso row is the giveaway to check for by hand in that unlikely case.

### `cif1.exe` (TOPAS's own bundled converter -- fallback / comparison only)

`cif1.exe` is a real, standalone command-line utility bundled with every TOPAS install (alongside `sgcom6.exe`, `tc.exe`, etc. -- see "Locating your TOPAS installation"). It does a straightforward field-by-field CIF→`str` conversion but is **not Wyckoff-aware** -- confirmed directly by the user (TOPAS-Academic's author) -- so it will not detect tied coordinates on a partial special position and will happily emit three independent free values for a site that actually needs a `Get()`-based constraint. Prefer `cif_to_str.py` above; fall back to `cif1.exe` only as a second opinion or when `cif_to_str.py`'s limitations above apply. `pdb_cif_to_str_file` (`references/18-protein-refinement.md`) is a *different* tool again: built specifically for protein-scale PDBx/mmCIF files from RCSB PDB (residue-per-rigid-body output like `SER_CA_1_5_ATOM`), not general small-molecule/inorganic CIFs.

Usage (needs the resolved TOPAS install dir -- see `python3 scripts/topas_install.py --inc-dir`, which prints the directory `cif1.exe` sits alongside):

```
cif1.exe input.cif output.inp
```

Confirmed via a real conversion (`test_examples/ceo2.cif`) and cross-checked by actually running the result through `tc.exe` (not just the heuristic checker) with zero errors:

```
str
    phase_name "?CeO2?"
    a 5.410278
    b 5.410278
    c 5.410278
    al 90
    be 90
    ga 90
    volume 158.3648
    space_group "F_M_3_M"
    site Ce1    x 0             y 0             z 0             occ Ce+4  1          beq 0.200
    site O1     x 0.25          y 0.25          z 0.25          occ O-2   1          beq 0.43
```

Two things worth knowing about this output before using it as-is:

- **`volume` is a real, valid TOPAS keyword** despite not appearing in `references/21-keyword-index.md`'s harvested set or any indexed manual chapter -- confirmed by running the raw `cif1.exe` output through `tc.exe` directly with no error (`Num independent parameters: 0`, clean `Rwp` computed). A gap in this skill's own keyword coverage, not a bug in `cif1.exe`'s output. `space_group` and `phase_name` being quoted strings (`"F_M_3_M"`, `"?CeO2?"`) rather than bare tokens is also valid -- both forms are documented (see the `$`-sigil single-string-arg check in `check_inp_syntax.py`).
- **`phase_name` can inherit garbage straight from an under-filled CIF field.** Here it became the literal string `"?CeO2?"` -- the question marks are CIF's own "value not specified" placeholder convention (from `_chemical_name_mineral ?CeO2?` in the source CIF), carried through verbatim rather than treated as absent. Check `phase_name` after every `cif1.exe` conversion and clean it up by hand (e.g. to a plain identifier like `ceo2_cif`) rather than trusting it blindly.

`cif1.exe`'s output is deliberately generic: explicit `a`/`b`/`c`/`al`/`be`/`ga` rather than a symmetry-shorthand macro like `Cubic(...)`, and no `scale` at all (that's left for you to add, seeded appropriately for the refinement at hand) -- expected, not a bug, since the tool has to handle every crystal system uniformly and can't know your intended scale.

## Updating `references/paper-summaries.md` from new algorithm papers (PDFs)

If someone points you at a folder of the actual journal-paper PDFs behind TOPAS's algorithms and asks you to read/summarize them (rather than the Technical Reference manual), here's what worked:

**PDF text extraction**: the `Read` tool's PDF-page-rendering path depends on `pdftoppm` (poppler-utils), which was not installed in this environment and errored on every PDF. The fallback that worked: `python -m pip install pypdf` (not preinstalled either), then extract per-page text with `PdfReader(path).pages[i].extract_text()`. This is plain-text extraction (no OCR/layout awareness), which is sufficient for reading a paper's prose/equations-as-text but will mangle any embedded math that was originally a typeset equation object, the same limitation already documented for the DOCX manual above. Extract to a scratch directory (`$CLAUDE_JOB_DIR/tmp` in a background job, `/tmp` otherwise), never into the skill itself — the whole point of `paper-summaries.md` is to hold original summaries plus a citation, not copies of the papers.

**Don't assume filenames map 1:1 to distinct papers.** In one real folder of 28 PDFs: one file was an IUCr *oral presentation certificate*, not a paper at all (worth a quick first-page check before spending time on it); two differently-named files (`band-flipping.pdf` and `charge-flipping-neutron-aj5281.pdf`) turned out to be a pre-typesetting draft and the final published version of the *same* paper; and two other differently-named files (`DSLR-3.pdf`, `fuel-cells.pdf`) turned out to be exact duplicate copies of one paper. Read at least the title/author/abstract of every file before writing or trusting a citation — a plausible-looking filename is not a substitute.

**A paper's own header is the authoritative citation, and it can catch existing errors.** IUCr/Acta Cryst electronic reprints print `J. Appl. Cryst. (YYYY). VOL, PAGES` directly on the cover page and again in the article header — reading it directly caught two real citation bugs already sitting in `paper-summaries.md`: a page range off by 3 (502–509 written down for what the paper itself says is 502–512), and a wrong journal/volume guess (a paper recorded as "Acta Cryst. A72 / related manuscript" that was actually never published there — it's *J. Appl. Cryst.* 49, 1806–1809). Don't trust a citation already in the file just because it's already there; re-derive it from the source when the source is available.

**Summaries should carry the paper's concrete numbers, not just its abstract-level claim.** A first pass tends to produce an accurate but thin summary ("describes a fast method for X, Y% faster"); a full read of the body text usually turns up specific default parameter values, stated accuracy/correlation caveats, and worked-example numbers that are exactly the kind of thing someone debugging a `.inp` script needs (e.g. "these two parameters are >90% correlated, fix one" or "this default constant is 0.5, reduce it if X"). These are worth a second pass even when the first-pass summary isn't wrong, just shallow.

## Updating this skill from a revised Technical Reference (DOCX or PDF)

These reference files were built from *TOPAS-Academic Technical Reference* (Alan Coelho), cross-checked against both a PDF export and the source `.docx`. If a newer edition shows up in a future session, this section is the accumulated playbook for updating the skill from it — without needing to re-derive any of this from scratch.

**Get the file open correctly.** For a large `.docx`, don't blindly `unzip` the whole archive — media/fonts can be tens of megabytes and slow things down. Pull just `word/document.xml` (`unzip -p file.docx word/document.xml > document.xml`) for text/structure work; only unpack fully if you need to edit and repackage. Parse with `lxml.etree` and the WordprocessingML namespace, not regex. Regex against raw OOXML has a specific, real failure mode: a self-closing `<w:t/>` element makes a naive `<w:t[^>]*>(.*?)</w:t>` pattern swallow everything up to the *next* real closing tag, silently leaking raw XML into what should be clean text. This happened mid-session and inflated a false-positive error count until it was caught and redone with lxml.

**Re-resolve chapter/section numbers by heading text, not by number.** Chapter numbering drifts between editions and even between exports of what's nominally the same edition — this session hit a real case where a duplicated "Indexing" chapter existed in one copy but had one of its two copies removed in another, shifting every subsequent chapter number down by one. Walk all `Heading1`–`Heading4`-styled paragraphs, rebuild a running `chapter.section.subsection` counter per level, and match by the heading's actual text (e.g. "GUI Functionality"), not by an assumed number. The skill's own `00`–`24` filename prefixes are an independent organizational scheme created when this skill was built — they intentionally do not track the manual's own chapter numbers, and shouldn't be made to.

**Keywords are marked by a Word character style, not just a color.** Style `AacKeyword1` renders orange (`#984806`, italic); style `AacKeywordHyperLink` renders blue (`#0032FF`, bold+italic) for keyword mentions that are also cross-reference links — about 7% of all keyword mentions use this variant. A color-only scan for orange will silently miss those. Macro names use `AacMacro` (navy `#002060`, italic); reserved parameter names use `AacReservedParameters` (italic, inherited color); file names use `AacFile` (`#262626`, bold+italic). One specific trap: a run can carry the `AacKeyword1` style tag while also carrying an explicit `<w:color w:val="auto"/>` override — this renders as ordinary black text despite the style tag, and is leftover formatting from an edit, not a real keyword instance. Filter these out rather than counting them.

**Reconstructing keyword-dependency hierarchy (the `Ttop`/`Tcomm_2`/etc. trees) requires indentation, and indentation hides behind two different mechanisms.** Some sections use paragraph-level `w:ind`/`@left` (in twips — roughly 357–360 ≈ one level, 714–720 ≈ two levels, 1068–1074 ≈ three levels); others use literal leading `<w:tab/>` elements inside the paragraph instead, with no `w:ind` at all. Check both for every paragraph and take whichever signal is non-zero. A bare `T`-prefixed name (`Ttop`, `Tcomm_2`, `TMinimization`, ...) starts a new type's own definition; when that same name appears as a *member* elsewhere, it means "insert everything that type defines here too" (a mixin/include), not a nested child object — unless it's genuinely a child object like `[str ...]` containing further bracketed keywords. Not every chapter has real nesting to lose: several (the whole "Energy Minimization" chapter, "Deconvolution") are only ever one level deep, and flattening those into a single line loses nothing — check actual indent values before assuming a fix is needed, rather than reflexively rebuilding every keyword block into a tree. Also don't mistake ordinary code-example indentation (a macro body, a C++ snippet) for a keyword hierarchy — those are just code formatting, not type-dependency structure.

**Verify any suspected content difference or typo two independent ways before reporting it as real.** Cross-check what the DOCX shows against an independent extraction of the PDF (or vice versa) before concluding something is a genuine manual error rather than an artifact of one particular extraction path. Confirmed real findings from this process (fixed in these references): `out_dependents_for` → `out_dependences_for`; a doubled closing parenthesis in an `index_x0` equation; a missing closing parenthesis in a `cloud_try_accept` example; a missing terminating semicolon in a `val_on_continue` example; `ATMSCAT.CPP`/`NEUTSCAT.CPP` → the manual's actual `ATMSCAT.TXT`/`NEUTSCAT.TXT`. Conversely, don't "fix" the manual's own deliberate conventions — `...` meaning "content omitted here for brevity" is not incomplete or broken code, and reproducing it faithfully (including in the skill's reference copies) is correct.

**`scripts/check_inp_syntax.py` can run at scale across many extracted snippets at once, but expect real false positives**, from: the `...` omission convention tripping brace/equation-termination checks; non-INP content that happens to contain brackets or parens (NDX output-file listings, numbered results tables, space-group symbols like `P42/nmc`); and ordinary English words in comments coincidentally resembling a real keyword name. Hand-verify every flag against the source before reporting it — don't forward raw checker output as findings.

**This environment has occasionally reverted a file to an earlier state between turns** — this happened at least twice in one session, once losing a specific fix and once losing an entire added section, with no other visible cause. After a long chain of edits, it's worth spot-checking that a sample of earlier fixes are still present (`grep` for the corrected text) before declaring the update complete, rather than assuming persistence.

**A full re-verification pass (not just a typo/error hunt) found a single dominant failure mode: a `##`/`###` heading was kept but its body prose was silently dropped**, while surrounding tables and worked-example code blocks survived untouched. This showed up at every scale — a whole subsection left completely empty under its own heading (`references/14-indexing.md`'s "Extinction subgroup determination", `references/13-rigid-bodies.md`'s "Rigid body parameter errors propagated to fractional coordinates", `references/22-charge-flipping.md`'s chapter-opening paragraph), a multi-branch conditional/enumerated list missing 1-2 of its branches (the `save_best_chi2`/`continue_after_convergence` 4-branch truth table in `references/03-minimization-and-convergence.md` had only 2 branches; several "N modes of operation" and "further tips" lists elsewhere were missing their first or last item), and a keyword's bracket signature present with no description sentence below it (`correct_for_temperature_effects`, `use_Fc`, `verbose`, `scale_F000` in `references/22-charge-flipping.md`; `point_by_point_beq_fo_etc` in `references/20-miscellaneous.md`). When re-verifying this skill again in future, grep each chapter's paragraph *count* against the source rather than only checking for specific known strings — a dropped paragraph produces no error, just silence. Also: the `ATMSCAT.CPP`/`NEUTSCAT.CPP` → `ATMSCAT.TXT`/`NEUTSCAT.TXT` fix noted above **regressed** back to `.cpp` in `references/20-miscellaneous.md` by the time of this pass, confirming the file-reversion issue is a real, recurring risk and not a one-off — spot-check past confirmed fixes, don't assume they stayed fixed.
